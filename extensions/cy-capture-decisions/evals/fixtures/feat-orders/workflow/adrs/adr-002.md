# ADR-002: Event-sourcing for orders

- Status: Accepted
- Date: 2026-07-11

## Context

Orders must support audit and point-in-time replay for dispute resolution and analytics. Any future
feature that reads or reacts to order history depends on how order state is persisted.

## Decision

Persist orders as an append-only event stream; derive current state by folding events. Publish the
events on the internal bus so other features can subscribe.

## Alternatives Considered

- Mutable rows with an audit-log side table (rejected: replay is lossy, audit drifts from truth).
- CRUD with soft deletes (rejected: no ordered history for replay).

## Consequences

- Enables audit + replay for every downstream consumer (cross-feature-durable).
- Read models must be projected; writers must be idempotent on the stream.
