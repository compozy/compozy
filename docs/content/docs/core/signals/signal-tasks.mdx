## Overview

Signal tasks are the **event publishers** in Compozy's workflow orchestration system, enabling sophisticated publish-subscribe communication patterns. They dispatch structured messages that can trigger wait tasks, start workflows via [signal triggers](/docs/core/signals/signal-triggers), and coordinate complex multi-system operations.

## Quick Start

<Steps numbered>
<Step title="Define a signal task" description="Create your first signal emitter">

```yaml
- id: send-notification
  type: signal
  signal:
    id: "user-approved"
    payload:
      user_id: "{{ .workflow.input.user_id }}"
      approved_by: "{{ .workflow.input.approver }}"
      timestamp: "{{ now }}"
```

Learn more about [template expressions](/docs/core/yaml-templates/context-variables) and [Sprig functions](/docs/core/yaml-templates/sprig-functions).

</Step>
<Step title="Create a receiver" description="Set up a workflow to receive signals">

Configure a workflow with a wait task or [signal trigger](/docs/core/signals/signal-triggers):

```yaml
# Option 1: Wait task
- id: wait-for-approval
  type: wait
  wait_for: "user-approved"
  on_timeout: handle-timeout

# Option 2: Signal trigger
triggers:
  - type: signal
    name: user-approved
```

</Step>
<Step title="Test the flow" description="Execute and monitor signal delivery">

Use the [CLI workflow commands](/docs/cli/workflow-commands) to test your signal flow and monitor execution.

</Step>
</Steps>

### Signal ID Strategies

<Tabs items={["Static IDs", "Dynamic IDs", "Context-Aware IDs", "Hierarchical Patterns"]}>
<Tab value="Static IDs">

Use static IDs for consistent system-wide events:

```yaml
signal:
  id: "system-maintenance"
  id: "daily-batch-complete"
  id: "deployment-finished"
```

**Best for**: System notifications, scheduled workflows, infrastructure events

</Tab>
<Tab value="Dynamic IDs">

Create targeted signals using workflow input:

```yaml
signal:
  id: "user-{{ .workflow.input.user_id }}-action"
  id: "order-{{ .workflow.input.order_id }}-status"
  id: "tenant-{{ .env.TENANT_ID }}-event"
```

**Best for**: User-specific events, entity updates, multi-tenant patterns

</Tab>
<Tab value="Context-Aware IDs">

Build fully dynamic routing with [template expressions](/docs/core/yaml-templates/overview):

```yaml
signal:
  id: "{{ .workflow.input.event_type }}-{{ .workflow.input.entity_id }}"
  id: "{{ .task.output.signal_name }}"
  id: "{{ .env.ENVIRONMENT }}-{{ .workflow.input.signal_type }}"
```

**Best for**: [Router task integration](/docs/core/tasks/router-tasks), runtime-determined events

</Tab>
<Tab value="Hierarchical Patterns">

Build scalable namespacing with hierarchical signal organization:

```yaml
signal:
  id: "{{ .env.TENANT }}.{{ .workflow.input.domain }}.{{ .workflow.input.entity }}.{{ .workflow.input.action }}"
  # Examples:
  # "acme.orders.payment.completed"
  # "beta.users.profile.updated"
  # "prod.inventory.stock.depleted"
```

**Best for**: Multi-tenant architectures, microservice coordination, enterprise-scale event routing

</Tab>
</Tabs>

## Common Patterns

### Event Notification

Send workflow completion notifications:

```yaml
- id: notify-completion
  type: signal
  signal:
    id: "workflow-completed"
    payload:
      workflow_id: "{{ .workflow.id }}"
      completed_at: "{{ now }}"
      status: "success"
```

Perfect for monitoring integration and triggering downstream workflows.

### Approval Workflows

Implement human-in-the-loop patterns with the [Event API](/docs/core/signals/event-api):

```yaml
- id: request-approval
  type: signal
  signal:
    id: "approval-requested"
    payload:
      request_id: "{{ .workflow.input.request_id }}"
      requester: "{{ .workflow.input.user_id }}"
      amount: "{{ .workflow.input.amount }}"
      approval_link: "{{ .env.APP_URL }}/approve/{{ .workflow.input.request_id }}"
```

See complete approval workflow patterns in the patterns documentation.

### Data Pipeline Coordination

Coordinate multi-stage processing with [collection tasks](/docs/core/tasks/collection-tasks):

```yaml
- id: data-stage-complete
  type: signal
  signal:
    id: "pipeline-{{ .workflow.input.pipeline_id }}-stage-{{ .workflow.input.stage }}"
    payload:
      pipeline_id: "{{ .workflow.input.pipeline_id }}"
      stage: "{{ .workflow.input.stage }}"
      next_stage: "{{ .task.output.next_stage }}"
      output_location: "{{ .task.output.data_path }}"
```

## Advanced Patterns

### Conditional Signal Dispatch

Combine with [router tasks](/docs/core/tasks/router-tasks) for dynamic routing:

```yaml
- id: route-by-priority
  type: router
  routes:
    - condition: '.task.output.priority == "urgent"'
      next: send-urgent-signal
    - condition: '.task.output.priority == "normal"'
      next: send-normal-signal
    - default: send-low-priority-signal

- id: send-urgent-signal
  type: signal
  signal:
    id: "urgent-task-created"
    payload:
      task_data: "{{ .task.output }}"
      escalation_required: true
```

### Error Recovery Signals

Implement error handling patterns with notifications:

```yaml
- id: process-data
  type: basic
  tool:
    id: data-processor
  with:
    data: "{{ .workflow.input.data }}"

  on_error:
    next: send-error-signal

- id: send-error-signal
  type: signal
  signal:
    id: "processing-error"
    payload:
      workflow_id: "{{ .workflow.id }}"
      error: "{{ .tasks.process-data.error }}"
      timestamp: "{{ now }}"
      severity: "high"
      retry_recommended: true
```
