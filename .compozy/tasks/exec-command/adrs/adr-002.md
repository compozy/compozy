# ADR-002: Replace `.tmp/codex-prompts` with Workspace-Scoped Run Artifacts

## Status

Accepted

## Date

2026-04-05

## Context

Current prompt and log artifacts are written under `.tmp/codex-prompts/<label>/`. This path is legacy in three ways:
- It is hardcoded in planner input resolution instead of being modeled as shared runtime policy.
- It is named after Codex even though the runtime supports multiple ACP agents.
- It behaves like a temporary scratch directory even though the generated prompts and logs are valuable debugging and automation artifacts.

The new `compozy exec` command also needs a stable place for resolved prompts, logs, and machine-readable result files. Extending `.tmp/codex-prompts` would preserve a naming and ownership bug instead of fixing it.

## Decision

Retire `.tmp/codex-prompts` and move all execution artifacts to a unified workspace-scoped root under `.compozy/runs/<run-id>/`.

Each run directory will contain:
- Run-level metadata such as `run.json`
- A `jobs/` subdirectory with prompt and log files per job
- Optional machine-readable result files such as `result.json` when JSON output is requested

This layout will be shared by existing directory-backed runs and the new ad hoc `exec` mode.

## Alternatives Considered

### Alternative 1: Keep `.tmp/codex-prompts` and add `exec` artifacts there

- **Description**: Continue using the existing path for all prompt/log files.
- **Pros**: Lowest migration cost.
- **Cons**: Preserves codex-specific naming, hides important runtime artifacts in a temp-like path, and keeps artifact policy hardcoded in the wrong layer.
- **Why rejected**: It extends a known legacy mistake instead of correcting it.

### Alternative 2: Use `.compozy/runtime/runs/`

- **Description**: Store run artifacts under a more explicitly internal runtime directory.
- **Pros**: Clear separation from other workspace artifacts.
- **Cons**: Adds another top-level concept under `.compozy` without strong functional benefit.
- **Why rejected**: `.compozy/runs/` is simpler, easier to discover, and sufficient for the current scope.

## Consequences

### Positive

- Gives all execution modes one canonical artifact root.
- Makes run outputs reproducible and scriptable, especially for JSON output.
- Removes runtime naming that is tied to one agent vendor.

### Negative

- Existing tests and user expectations around the old path must be updated.
- Artifact retention may need explicit policy later if `.compozy/runs/` grows large.

### Risks

- If run IDs are not deterministic enough for tests, artifact paths can become hard to assert.
- If run metadata is underspecified, JSON consumers may still need to scrape logs.

## Implementation Notes

- Do not auto-migrate historical `.tmp/codex-prompts` directories in the first implementation.
- Introduce shared helpers for `RunsBaseDirForWorkspace` and per-run directory allocation so path policy is not hardcoded in the planner.
- Include run ID, execution mode, IDE, model, and output format in run-level metadata.
