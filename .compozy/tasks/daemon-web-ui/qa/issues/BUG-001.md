# BUG-001: Workflow inventory run start was missing and initially unverified

**Severity:** High
**Priority:** P1
**Type:** Functional
**Status:** Fixed

## Environment

- **Build:** `v0.1.12-25-g7cb682b`
- **OS:** macOS arm64
- **Browser:** Playwright Desktop Chrome / Headless Chrome 147
- **URL:** `/workflows`

## Summary

During `task_16` QA, a real operator could not start a workflow run from the daemon-served workflow inventory because the browser surface had no run-start control. Once the UI action was added, the first browser request path still failed because the route sent an unsupported presentation mode, and the Playwright smoke harness also needed a non-conflicting seeded run to validate the public flow cleanly.

## Reproduction

```bash
# Initial QA inspection
rg -n "useStartWorkflowRun|startWorkflowRun" web/src

# Focused browser smoke after wiring the route
bunx playwright test --config web/playwright.config.ts -g 'starts a workflow run from the workflow inventory'

# Direct daemon request that exposed the unsupported presentation mode during debugging
curl -sS -D .compozy/tasks/daemon-web-ui/qa/logs/16-run-start-curl-headers.txt \
  -o .compozy/tasks/daemon-web-ui/qa/logs/16-run-start-curl-body.json \
  -X POST "<daemon-base-url>/api/tasks/daemon-web-ui/runs" \
  -H 'Content-Type: application/json' \
  -H 'X-Compozy-Workspace-ID: <workspace-id>' \
  --data '{"workspace":"<workspace-id>","presentation_mode":"text"}'
```

Observed before the fix:

- The workflow inventory exposed no browser run-start control for `daemon-web-ui`.
- The first browser POST path used an unsupported `presentation_mode` and the daemon rejected it.
- The first focused smoke rerun still failed with `409` because `web/e2e/global.setup.ts` pre-seeded a live `daemon-web-ui` task run, so the smoke test collided with an already-active run.

## Expected

An operator should be able to open `/workflows`, click `Start run` for an active workflow, receive a `201 Created` response from the daemon, and see a success banner with a link to the new run.

## Root cause

The browser operator flow was incomplete across both product and QA harness layers:

- `@/systems/runs` already exposed the `/api/tasks/{slug}/runs` POST contract, but `/workflows` had no UI wiring for a real operator to trigger it.
- The initial route implementation sent a presentation mode the daemon does not accept for browser task starts.
- The Playwright fixture seeded a standalone `daemon-web-ui` task run, which made the new run-start smoke test fail with a harness-only conflict before the public flow could be validated.

## Fix

- Wired `/workflows` to `useStartWorkflowRun` and rendered a workflow-level `Start run` action plus success banner/link.
- Updated the browser request to use `presentation_mode: "detach"`.
- Added durable regression coverage in `web/src/routes/-workflow-tasks.integration.test.tsx`, `web/src/systems/workflows/components/workflow-inventory-view.test.tsx`, and `web/e2e/daemon-ui.smoke.spec.ts`.
- Moved the Playwright seeded standalone task run from `daemon-web-ui` to `daemon` in `web/e2e/global.setup.ts` so the smoke flow no longer collides with itself.

## Verification

- `bunx vitest run --config vitest.config.ts web/src/routes/-workflow-tasks.integration.test.tsx web/src/systems/workflows/components/workflow-inventory-view.test.tsx`
- `bunx playwright test --config playwright.config.ts -g 'starts a workflow run from the workflow inventory'`
- `make verify`

## Impact

- **Users Affected:** Browser operators using the daemon web UI
- **Frequency:** Always before the fix
- **Workaround:** Use CLI task execution directly instead of the browser inventory
