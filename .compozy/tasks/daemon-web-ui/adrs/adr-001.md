# ADR-001: Mirror AGH's Runtime Frontend Topology with `web/` and `packages/ui`

## Status

Accepted

## Date

2026-04-20

## Context

The daemon web UI must follow the same frontend package structure and library stack used in `/Users/pedronauck/dev/compozy/agh`, but the application itself must implement the local `docs/design/daemon-mockup` experience.

The repository currently has no JS workspace, no `web/` runtime app, and no shared frontend UI package. The user explicitly asked to use the same package structure, shadcn setup, Tailwind, Vite, Vitest, and related frontend libraries used in AGH, while avoiding unrelated workspaces that do not support the daemon UI directly.

## Decision

Loopy will add only the AGH runtime-facing frontend slices:

- `web/` for the daemon-served React SPA
- `packages/ui/` for shared UI primitives, tokens, and composition helpers

The implementation will mirror AGH's runtime frontend layout and toolchain, but it will not introduce broader AGH workspaces such as `packages/site` unless a later feature requires them.

## Alternatives Considered

### Alternative 1: Build the UI directly inside one `web/` app without a shared package

- **Description**: Keep all components and styling local to one SPA package.
- **Pros**: Lower setup cost and fewer workspace edges.
- **Cons**: Diverges from AGH, duplicates primitives, weakens reuse, and makes it harder to preserve the same frontend architecture across products.
- **Why rejected**: The user explicitly wants the same structural model used in AGH.

### Alternative 2: Mirror AGH's broader monorepo layout immediately

- **Description**: Reproduce `packages/site` and other AGH workspaces up front.
- **Pros**: Maximum structural parity with AGH.
- **Cons**: Adds unused workspaces, more tooling overhead, and design noise unrelated to the daemon UI.
- **Why rejected**: The user explicitly chose the runtime-only mirror, not the full monorepo expansion.

## Consequences

### Positive

- Frontend implementation can reuse proven AGH package boundaries and tooling.
- Shared UI primitives stay out of route-level feature code.
- The repo gains a clean path for future frontend reuse without dragging in docs-site concerns.

### Negative

- The repository must adopt a JS workspace and frontend build pipeline where none exists today.
- Tailwind scanning, linting, and workspace tooling must be kept aligned across two packages.

### Risks

- Partial AGH parity could drift if the shared package contract is copied loosely instead of mirrored deliberately.
- Introducing the workspace without clear boundaries could let `web/` bypass `packages/ui` and reintroduce duplication.

## Implementation Notes

- Mirror AGH's runtime-facing conventions for `web/src/routes`, `web/src/components`, and `web/src/systems/<domain>/`.
- Keep `packages/ui/` as the shared source for reusable primitives and styling tokens.
- Do not introduce `packages/site` in this feature.
