# ADR-003: Use Daemon-Only REST/SSE Contracts with OpenAPI-Generated Web Types

## Status

Accepted

## Date

2026-04-20

## Context

The user selected a strict daemon-only integration boundary for the web UI. The browser must not read workspace files directly for spec, memory, task, review, or run detail views. All reads and mutations must go through daemon-owned contracts.

The user also selected AGH's typed contract model, where the frontend consumes generated API types instead of maintaining hand-written REST types.

## Decision

The daemon web UI will consume daemon-owned REST and SSE APIs exclusively. OpenAPI will define the canonical web contract, and the frontend will use generated types and a typed client as its primary integration surface.

This applies to:

- workflow inventory and detail reads
- run list, snapshots, event streams, and cancellations
- review rounds, issues, and review-fix actions
- spec and memory document reads
- operational actions such as sync, archive, and run start

## Alternatives Considered

### Alternative 1: Hybrid browser access to workspace artifacts

- **Description**: Use daemon APIs for operations, but let the browser read some files directly.
- **Pros**: Lower backend lift for document-heavy views.
- **Cons**: Breaks the daemon as the control plane, weakens security and consistency, and creates divergent data paths.
- **Why rejected**: The user explicitly rejected hybrid access.

### Alternative 2: Hand-written frontend types and clients

- **Description**: Keep REST endpoints but write and maintain web types manually.
- **Pros**: Lower upfront codegen work.
- **Cons**: Higher drift risk, weaker refactor safety, and less parity with AGH.
- **Why rejected**: The user explicitly selected the OpenAPI-typed model.

## Consequences

### Positive

- The daemon remains the single source of truth for both UI and CLI surfaces.
- Web integrations become refactor-safe and contract-driven.
- SSE flows stay aligned with the existing daemon transport architecture.

### Negative

- The daemon must add richer document and dashboard endpoints that do not yet exist.
- OpenAPI generation becomes part of the frontend workflow and CI path.

### Risks

- If SSE behavior is underspecified in the contract, the frontend may still drift in stream handling.
- Partial codegen adoption could leave some endpoints typed and others manual, weakening the contract.

## Implementation Notes

- Extend the daemon API to serve spec, memory, workflow overview, and review detail payloads required by the mockup.
- Keep the browser free of filesystem assumptions.
- Define clear snapshot and stream contracts for long-running views.
