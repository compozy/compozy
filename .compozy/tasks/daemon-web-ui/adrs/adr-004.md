# ADR-004: Scope V1 to Operational and Rich Read Surfaces, Not In-Browser Authoring

## Status

Accepted

## Date

2026-04-20

## Context

The mockup clearly describes an operator console: dashboard, workflows, runs, reviews, memory, and spec navigation. It also includes operational actions such as sync, archive, and run control. The user selected a v1 scope that is stronger than read-only observability, but narrower than a full browser IDE or artifact editor.

## Decision

V1 will ship:

- dashboard and daemon health views
- workflow inventory and workflow detail surfaces
- run list, run detail, and live watch views
- review list/detail and review-fix dispatch flows
- spec and memory read views
- operational mutations: sync, archive, run start, run cancel, and related control actions

V1 will not ship browser-based editing for PRDs, TechSpecs, tasks, or memory files.

## Alternatives Considered

### Alternative 1: Read-only observability UI

- **Description**: Restrict the web UI to dashboards and inspection.
- **Pros**: Lower scope and lower daemon API expansion.
- **Cons**: Fails to deliver the operator-console model implied by the mockup.
- **Why rejected**: The user chose operational + rich read scope.

### Alternative 2: Full browser authoring from day one

- **Description**: Add create/edit flows for workflow artifacts and memory.
- **Pros**: Maximizes browser capability.
- **Cons**: Much larger scope, more editor-state complexity, and more conflict resolution work between browser and filesystem flows.
- **Why rejected**: The user explicitly limited v1 to operational + read surfaces.

## Consequences

### Positive

- V1 stays aligned with the mockup's operator-console value without dragging in browser editing complexity.
- Backend work can focus on operational and read models first.

### Negative

- Some navigation surfaces will remain read-only even when the browser can inspect the underlying artifact.
- The UI must clearly distinguish action surfaces from document-authoring surfaces.

### Risks

- If view affordances imply editing that is not implemented, the UX may feel inconsistent.
- Review-fix and run-start flows need strong status feedback so the UI still feels operationally complete.

## Implementation Notes

- Keep spec and memory pages intentionally read-only in v1.
- Provide explicit action affordances only for supported operational mutations.
- Avoid placeholder editor shells that suggest unsupported authoring flows.
