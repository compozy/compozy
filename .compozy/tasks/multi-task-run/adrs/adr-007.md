# ADR-007: Use One Task-Multi Scheduler with Worktree-Owned Child Runs

## Status

Accepted

## Date

2026-06-14

## Context

ADR-005 accepts true parallel multi-run execution for
`compozy tasks run --multiple`, using git worktrees to isolate child task runs.
The existing daemon implementation already has a `task_multi` parent run,
ordered child items, parent events, event-derived snapshots, and a tabbed
attach UI. It currently implements only enqueued execution: all items are
queued, then one child run starts and finishes before the next child starts.

Parallel execution introduces three technical choices that affect the same
surface:

- whether to add a separate parallel coordinator or evolve the current
  sequential coordinator into a scheduler
- whether child runs should be registered under the original workspace or under
  each physical worktree root
- where to store per-child worktree metadata needed by snapshots, attach, and
  final handoff

The run manager stores each run under one `workspace_id`, while runtime
execution uses `RuntimeConfig.WorkspaceRoot` and task paths derived from that
root. Sync and extension child-run paths also expect the workspace root to
match the filesystem tree being executed. Keeping the database workspace on the
original checkout while executing inside a worktree would split run identity
from runtime behavior.

## Decision

The daemon will model enqueued and parallel multi-run execution as branches of
one `task_multi` scheduler.

The scheduler will:

- keep the existing `task_multi` parent run mode, endpoints, parent event
  stream, snapshot endpoint, and tabbed attach surface
- dispatch the current enqueued behavior through an `enqueued` branch
- dispatch true parallel behavior through a bounded `parallel` branch
- share item state transitions, terminal aggregation, cancellation, and
  snapshot reconstruction rules across both branches where practical

For parallel mode, each child run will be registered against the workspace row
resolved from its physical worktree root. The child runtime config will also
use that worktree root and the task directory inside it. The original
`task_multi` parent remains registered against the original workspace and links
children through `parent_run_id` plus parent multi-run events.

Worktree metadata will be persisted in the existing `task.multi.*` parent event
payloads and reconstructed into `TaskRunMultipleSnapshot`. The payload and
snapshot item model will include the worktree path, base branch, base commit,
and preservation status needed for UI and handoff.

## Alternatives Considered

### Alternative 1: Add a separate parallel coordinator

- **Description**: Leave the sequential coordinator mostly intact and add a
  distinct parallel coordinator under the same parent model.
- **Pros**: Smaller local changes to the current sequential code path.
- **Cons**: Duplicates lifecycle concepts that must stay identical for events,
  snapshots, cancellation, and terminal aggregation.
- **Why rejected**: The user selected one scheduler with explicit mode
  branches so shared behavior remains visibly shared.

### Alternative 2: Register child runs under the original workspace

- **Description**: Keep parent and child rows under the same workspace while
  remapping only the child runtime config to the worktree path.
- **Pros**: Workspace-filtered run lists show the parent and children together.
- **Cons**: Database workspace identity no longer matches task discovery, sync,
  watcher, or extension execution paths.
- **Why rejected**: Runtime correctness is more important than preserving the
  existing workspace-list grouping shape. Parent snapshots remain the grouping
  source for multi-run children.

### Alternative 3: Add a dedicated worktree metadata table

- **Description**: Store child worktree details in a new `global.db` table and
  use events only for live UI updates.
- **Pros**: More queryable durable state and easier direct recovery queries.
- **Cons**: Adds a migration and second source of truth for the existing
  event-sourced multi-run snapshot model.
- **Why rejected**: V1 should extend the existing parent event and snapshot
  architecture before adding new storage surfaces.

## Consequences

### Positive

- The daemon keeps one user-facing multi-run model across enqueued and parallel
  execution.
- Child task runs execute with workspace identity aligned to their real
  filesystem roots.
- Snapshots, attach, stream output, and final handoff can reuse the existing
  event-derived state model.
- Parent and child linkage stays explicit through `parent_run_id`.

### Negative

- Normal workspace-filtered run lists may show the parent under the original
  workspace and child runs under their worktree workspaces.
- Snapshot reconstruction must preserve enough worktree metadata in parent
  events before a child starts.
- The scheduler needs careful shared terminal aggregation so enqueued behavior
  does not regress while parallel mode becomes fail-late.

### Risks

- **Split observability**: Users may look at original-workspace run lists and
  not see worktree-owned child rows. Mitigate by making the multi-run snapshot,
  TUI, stream output, and final summary the canonical child view.
- **Event payload drift**: New worktree fields must stay backward compatible.
  Mitigate with additive JSON fields and payload compatibility tests.
- **Recovery gaps**: If metadata is emitted too late, daemon interruption can
  leave unreported worktrees. Mitigate by emitting allocation metadata before
  child launch.

## Implementation Notes

- Keep the dedicated multi-run API routes.
- Update `TaskRunMultiplePayload` and `TaskRunMultipleItem` additively.
- Register each parallel child workspace by resolving the worktree root through
  the existing workspace registry path.
- Remap both `RuntimeConfig.WorkspaceRoot` and `RuntimeConfig.TasksDir` before
  calling `startRun` for a parallel child.
- Do not reuse the existing worktree fingerprint package as the allocator.
