# ADR-005: Ship Worktree-Backed Parallel Multi-Run as an Opt-In Mode

## Status

Accepted

## Date

2026-06-13

## Context

`compozy tasks run --multiple <slug-a>,<slug-b>` already has a parent
`task_multi` run model, child task runs, tabbed attach state, and
`[tasks.run] run_multiple_mode` with accepted values `enqueued` and `parallel`.
Current behavior still downgrades configured `parallel` to enqueued execution
and the daemon rejects direct parallel mode because multiple agents editing the
same checkout can overwrite each other or contend on git state.

The user now wants true parallel execution for independent tasks in one task
bundle. The primary value is faster batches for a solo developer. V1 should be
minimal: explicit `--parallel` and config support, isolated git worktrees, and
aggregate status. If one task fails, sibling tasks should continue and the
parent should report aggregate failure after all children finish.

Research and council review reinforced the same constraint: git worktrees are
the right filesystem isolation primitive because each child gets its own
working tree, `HEAD`, and index while sharing repository objects. They are not a
complete sandbox. Ports, credentials, provider limits, caches, daemon state,
task artifacts, and merge/recombination conflicts still need explicit product
boundaries.

## Decision

V1 will add true parallel multi-run execution as an explicit, opt-in scheduling
mode for `compozy tasks run --multiple`.

Parallel mode will:

- be enabled by `--parallel` or `[tasks.run] run_multiple_mode = "parallel"`
- keep the existing parent `task_multi` run and child task-run model
- allocate one git worktree per child task run before launching the child
- run child task runs concurrently up to a bounded concurrency limit
- give each child a remapped workspace root and task directory inside its
  worktree
- keep per-child status visible through existing multi-run events, snapshots,
  and tabs
- let siblings continue when one child fails
- finish the parent as failed if any child failed, canceled, or could not start
- leave merge/recombination as an explicit manual step for V1

Parallel mode is an isolation-and-aggregation feature, not a general scheduler.
Compozy should document that worktrees isolate git working state but do not
isolate all local runtime resources.

## Alternatives Considered

### Alternative 1: Keep `parallel` as an enqueued fallback

- **Description**: Continue accepting `parallel` in config while printing the V2
  fallback message and running sequentially.
- **Pros**: No new worktree lifecycle, no new concurrent child behavior, and
  lowest implementation risk.
- **Cons**: The config value does not do what it says, which weakens user trust
  and leaves the fastest independent-batch workflow unsolved.
- **Why rejected**: The user explicitly wants `parallel` to run in parallel now,
  with git worktree isolation.

### Alternative 2: Run children concurrently in the current checkout

- **Description**: Start all child task runs at once using the same workspace
  root.
- **Pros**: Smallest possible scheduler change.
- **Cons**: Agents can overwrite each other's edits, corrupt assumptions by
  reading partially modified files, and contend on git state.
- **Why rejected**: It violates the core safety requirement for parallel coding
  agents.

### Alternative 3: Build a full merge-and-review platform

- **Description**: Add worktree creation, conflict prediction, auto-merge,
  branch review, cleanup dashboards, and dependency-aware scheduling in one
  feature.
- **Pros**: More complete parallel-agent product surface.
- **Cons**: Too large for the immediate user goal and likely to delay the
  fastest learning loop.
- **Why rejected**: V1 should prove that isolated parallel batches are valuable
  before Compozy owns recombination workflows.

## Consequences

### Positive

- Independent task bundles can complete in less wall-clock time.
- The already documented `parallel` mode becomes truthful.
- Existing parent/child run accounting, snapshots, and tabs remain the user
  surface instead of introducing a separate orchestration model.
- Worktree isolation prevents same-checkout file and git-index collisions.

### Negative

- Compozy must own durable worktree lifecycle state: path, branch, base commit,
  child workspace root, cleanup status, and recovery after daemon interruption.
- Child runtime configuration can no longer assume one shared workspace root.
- The parent coordinator needs a distinct parallel execution topology because
  sequential ordering, first-failure cancellation, and one-workspace assumptions
  no longer apply.
- Successful child output still requires manual integration in V1.

### Risks

- **False independence**: Two tasks can still make semantically conflicting
  changes. Mitigate by documenting that V1 is for independent tasks and by
  preserving per-child branches for manual review.
- **Shared runtime collisions**: Ports, local databases, caches, and provider
  limits remain shared. Mitigate with explicit limitations and future
  per-worktree runtime isolation hooks.
- **Orphaned worktrees**: Daemon crashes can leave directories behind. Mitigate
  by persisting worktree metadata before launch and adding cleanup/recovery
  affordances.
- **Unbounded fanout**: Starting too many agents can exhaust local or provider
  resources. Mitigate with a conservative concurrency cap.

## Implementation Notes

- Add a `--parallel` CLI flag as a per-invocation override for multi-run mode.
- Preserve `run_multiple_mode = "enqueued"` as the default.
- Treat config `parallel` and CLI `--parallel` as the same scheduling mode, with
  the CLI flag taking precedence for the invocation.
- Implement the daemon parallel path as a separate coordinator under the
  existing `task_multi` parent run model.
- Add a git worktree allocator instead of reusing the current
  `internal/core/run/internal/worktree` snapshot package, which only
  fingerprints workspace changes.
- Persist child worktree metadata before each child starts so snapshots and
  cleanup can survive detach/restart scenarios.
- Do not auto-merge, auto-push, or delete changed worktrees in V1.
