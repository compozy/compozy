# ADR-008: Make the Parallel Multi-Run Limit Configurable

## Status

Accepted

## Date

2026-06-14

## Context

ADR-006 originally left the exact V1 parallel concurrency cap to the TechSpec
and framed user-configurable concurrency as future work. During TechSpec
clarification, the user selected a configurable cap with a default of `2`, and
then selected both config and CLI exposure.

Parallel multi-run fanout can consume local CPU, filesystem watchers, provider
quota, agent sessions, ports, and daemon capacity. A fixed cap is simple, but
local machines and task batches vary enough that a power user may need to lower
or raise the limit per workspace or per invocation.

## Decision

V1 will expose a configurable parallel multi-run limit with a default of `2`.

Configuration:

```toml
[tasks.run]
run_multiple_parallel_limit = 2
```

CLI:

```bash
compozy tasks run --multiple task_01,task_02 --parallel --parallel-limit 2
```

Precedence will be:

1. `--parallel-limit <n>` when provided
2. `[tasks.run] run_multiple_parallel_limit`
3. default `2`

The limit applies only to `run_multiple_mode = "parallel"` or invocations that
set `--parallel`. Enqueued mode remains one child at a time.

## Alternatives Considered

### Alternative 1: Fixed cap only

- **Description**: Hard-code a conservative cap such as `2` or `3`.
- **Pros**: Smallest config and CLI surface.
- **Cons**: Does not let users adapt to local machine capacity or provider
  constraints.
- **Why rejected**: The user explicitly selected configurability.

### Alternative 2: Config only

- **Description**: Add `run_multiple_parallel_limit` to TOML but no CLI
  override.
- **Pros**: Keeps the `tasks run` CLI smaller.
- **Cons**: Makes one-off tuning require editing config.
- **Why rejected**: The user selected both config and CLI override.

### Alternative 3: Reuse existing runtime concurrency

- **Description**: Reuse the existing runtime `--concurrent` option for
  multi-run child fanout.
- **Pros**: Avoids another numeric flag.
- **Cons**: Conflates per-run internal task concurrency with parent-level child
  run fanout.
- **Why rejected**: Multi-run fanout and child runtime concurrency are separate
  scheduling layers and should remain separately named.

## Consequences

### Positive

- Default behavior remains conservative and predictable.
- Users can tune per workspace or per invocation without changing unrelated
  runtime concurrency settings.
- The implementation can keep enqueued mode unchanged while using the limit as
  a parallel scheduler semaphore.

### Negative

- This expands V1 beyond the earlier fixed-cap PRD assumption.
- Adds one config field, one CLI flag, validation rules, docs, and tests.
- Higher user-selected limits can still overload local or provider resources.

### Risks

- **Misconfiguration**: Invalid or extreme values can make runs fail or starve.
  Mitigate with validation and documented positive-integer semantics.
- **Semantic confusion**: Users may confuse `--parallel-limit` with existing
  runtime `--concurrent`. Mitigate with clear help text and examples.

## Implementation Notes

- Add `TaskRunConfig.RunMultipleParallelLimit *int` with TOML key
  `run_multiple_parallel_limit`.
- Add an effective config helper that defaults an unset value to `2`.
- Reject zero and negative values in config and CLI input.
- Add `--parallel-limit` to `tasks run`; it should be valid only with
  `--multiple` and effective only for parallel mode.
- Include the resolved limit in the daemon request or runtime override surface
  used by `StartTaskRunMultiple`.
