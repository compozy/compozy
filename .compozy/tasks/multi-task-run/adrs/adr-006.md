# ADR-006: Use a Speed-First Parallel MVP for the PRD Scope

## Status

Accepted

## Date

2026-06-14

## Context

The updated idea and ADR-005 define true parallel multi-run execution for
`compozy tasks run --multiple` as opt-in, worktree-backed, bounded, and
aggregate-fail-late. During PRD clarification, the user selected faster
completion for independent task batches as the strongest V1 success promise.
The user also selected preserving all child worktrees for manual review, using a
conservative fixed concurrency cap whose exact number belongs in the TechSpec,
and showing each child worktree path plus final status at the end of a parallel
run.

The PRD needs to translate those decisions into product requirements without
expanding into merge automation, conflict prediction, cleanup policy matrices,
or a broader multi-agent workflow platform.

## Decision

The PRD will use the Speed-First Parallel MVP approach.

V1 will focus on making independent task batches finish faster while preserving
trustworthy status and review handoff. The product scope will include:

- explicit `--parallel` and config-backed parallel mode
- preserved child worktrees for manual review
- clear child worktree path and final status output
- conservative concurrency, with the exact cap left to the TechSpec
- sibling continuation after child failure and aggregate parent failure

The PRD will treat richer review guidance, conflict hints, cleanup automation,
and configurable concurrency as future work unless they are required to explain
V1 behavior.

## Alternatives Considered

### Alternative 1: Review-Guided Parallel MVP

- **Description**: Include the speed-first scope plus suggested manual review
  order and richer integration guidance.
- **Pros**: Helps users decide what to inspect first after a parallel run.
- **Cons**: Pulls merge/review workflow concerns into the first product slice.
- **Why rejected**: V1 should prove faster independent batches before owning
  integration workflow design.

### Alternative 2: Managed Parallel Workflow

- **Description**: Add configurable concurrency, cleanup rules, review
  guidance, and conflict hints as part of the initial PRD.
- **Pros**: More complete product surface for power users.
- **Cons**: Expands the feature beyond the user's selected minimal scope and
  delays the core speed benefit.
- **Why rejected**: Too broad for V1 and better suited to a later product pass.

## Consequences

### Positive

- Keeps the PRD aligned with the user's selected success promise.
- Gives downstream TechSpec work a crisp MVP boundary.
- Avoids turning parallel execution into an integration or conflict-resolution
  product before the basic workflow proves value.

### Negative

- Users receive minimal integration guidance in V1.
- Cleanup and review-order decisions remain manual.
- The PRD will leave some product details for later phases or TechSpec
  refinement.

### Risks

- **Under-guided review handoff**: Mitigate by requiring path and status output
  for every child worktree.
- **Concurrency ambiguity**: Mitigate by requiring a conservative cap while
  leaving the exact number to the TechSpec.
- **Scope creep**: Mitigate by keeping conflict hints, auto-merge, and cleanup
  automation out of V1.

## Implementation Notes

- The PRD should describe user capabilities and outcomes, not implementation
  mechanics.
- The TechSpec should decide exact concurrency number, path naming, and cleanup
  mechanics.
