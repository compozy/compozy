# ADR-003: Detect Installation Method and Suggest Appropriate Upgrade Command

## Status

Accepted

## Date

2026-04-06

## Context

Compozy is distributed via four channels: Homebrew cask, NPM, `go install`, and direct binary download. Each channel has its own upgrade mechanism. Performing an in-place binary replacement on a Homebrew- or NPM-managed installation can break the package manager's state tracking, causing version mismatches and failed future upgrades.

## Decision

Detect the installation method at runtime and adapt the `compozy upgrade` behavior:

| Method | Detection | Behavior |
|--------|-----------|----------|
| Homebrew | Executable path under Homebrew prefix | Print `brew upgrade --cask compozy` |
| NPM | Executable path contains `node_modules` or npm global prefix | Print `npm install -g @compozy/cli@latest` |
| Go install | Executable path under `$GOPATH/bin` or `$GOBIN` | Print `go install github.com/compozy/compozy/cmd/compozy@latest` |
| Binary download | None of the above | Perform in-place self-update via go-selfupdate |

Only the "binary download" case performs automatic self-update. All package-manager cases print the correct command and exit.

## Alternatives Considered

### Alternative 1: Always self-update the binary directly

- **Description**: Ignore the installation method and always replace the binary in-place using go-selfupdate.
- **Pros**: Simplest implementation, one code path
- **Cons**: Breaks Homebrew/NPM version tracking. Users get version mismatches. Package managers may overwrite the update on next `brew upgrade`.
- **Why rejected**: Conflicts with package manager expectations and causes user confusion.

### Alternative 2: Only notify, never self-update

- **Description**: `compozy upgrade` only checks and prints instructions for all installation methods.
- **Pros**: Safest, no risk of breaking anything
- **Cons**: Poor UX for binary-download users who have no package manager to help them
- **Why rejected**: Binary-download users deserve a convenient in-place upgrade path.

## Consequences

### Positive

- Respects each package manager's conventions
- Binary-download users get convenient self-update
- Clear, actionable output for every installation method

### Negative

- Detection heuristics may fail on unusual setups (e.g., symlinked binaries)
- Must maintain detection logic as distribution channels evolve

### Risks

- False positive detection (e.g., binary in a path that happens to contain "node_modules"): mitigated by checking specific path patterns and falling back to self-update
- New distribution methods added later: mitigated by defaulting to self-update for unknown methods

## Implementation Notes

Detection uses `os.Executable()` to get the resolved binary path, then checks against known prefixes:
- Homebrew: `strings.Contains(path, "/Cellar/")` or `strings.HasPrefix(path, brewPrefix)`
- NPM: `strings.Contains(path, "node_modules")` or path under npm global prefix
- Go: path under `$GOPATH/bin` or `$GOBIN`
