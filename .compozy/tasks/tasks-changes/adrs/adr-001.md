# ADR-001: Task Metadata Schema v2 — Title in Frontmatter, Drop scope/domain

## Status

Accepted

## Date

2026-04-04

## Context

The current task file schema (`internal/core/model/model.go:181-188`) uses YAML frontmatter with `status`, `domain`, `type`, `scope`, `complexity`, `dependencies`. The human-readable task title lives only in the body as an H1 markdown header (`# Task 1: ACP Agent Layer & Content Model`). This has three problems:

1. **TUI cannot surface the task title.** The session timeline header only shows the static label `session.timeline` (`internal/core/run/ui_view.go:482`). To show "ACP AGENT LAYER & CONTENT MODEL" in the header when running `compozy start`, the title must be machine-readable.
2. **`scope` and `domain` are unused downstream.** `TaskEntry` fields `Domain` and `Scope` are parsed but never surfaced in the execution pipeline, TUI, or prompts. They are dead metadata.
3. **`type` is free-text today.** Values like "Feature Implementation" carry no semantic weight. The product direction (ADR-002) is to turn `type` into a constrained enum that drives future tooling (filtering, routing, UI styling), which requires a compact metadata contract.

## Decision

Introduce task metadata schema **v2** with these changes to `TaskFileMeta`:

- **Add** `title string` (required, non-empty)
- **Remove** `Domain string` and `Scope string`
- **Keep** `Status`, `TaskType` (as enum — see ADR-002), `Complexity`, `Dependencies`

The H1 body header (`# Task N: <title>`) is retained for human readability but the **frontmatter `title` is the canonical source** for all tooling. `validate-tasks` enforces that the frontmatter `title` matches the H1 body (trimmed, case-sensitive, ignoring the `Task N:` prefix).

## Alternatives Considered

### Alternative 1: Parse title from H1 body only, no frontmatter field

- **Description**: Extract `# Task N: <title>` via regex at parse time; no frontmatter change.
- **Pros**: Zero schema churn; no migration needed.
- **Cons**: Fragile regex; breaks if the author edits the H1 format (e.g., drops "Task N:", uses different casing). Title becomes undefined if the H1 changes.
- **Why rejected**: The TUI and validator need a deterministic contract; parsing unstructured body content is regression-prone.

### Alternative 2: Title in frontmatter only, drop the H1

- **Description**: Write title to frontmatter and remove the H1 from generated templates.
- **Pros**: Single source of truth; no sync logic.
- **Cons**: Markdown files lose their human-readable headline; diff reviews and GitHub renders become less useful.
- **Why rejected**: Task files are read by humans during review and in editors. Losing the H1 degrades readability for minimal tooling gain.

### Alternative 3: Keep `domain` and `scope` but repurpose them

- **Description**: Rename `scope` to something useful, keep `domain` as a category.
- **Pros**: Zero removal; backward-compatible.
- **Cons**: Overlaps with the new typed `type` field; multiplies categorization axes (domain + type + complexity) with no downstream consumer for the extras.
- **Why rejected**: YAGNI. Two taxonomic fields (`type` + `complexity`) are enough for the current roadmap.

## Consequences

### Positive

- TUI timeline header can display the task title directly from parsed metadata.
- `validate-tasks` has a clear, testable contract for title/H1 consistency.
- Task frontmatter shrinks from 6 fields to 5; simpler for LLM generation and human review.
- The H1/frontmatter sync rule is mechanical and catches editor mistakes.

### Negative

- Every existing task file must be migrated (handled by `compozy migrate`, see ADR-004).
- The `cy-create-tasks` skill template and all referenced examples need updating.
- Authors must maintain two locations (frontmatter + H1) for the title; mitigated by the validator and by the skill generating both.

### Risks

- **Risk**: Users hand-edit the H1 and forget the frontmatter, desync silently.
  **Mitigation**: `validate-tasks` runs automatically before `compozy start` (ADR-003) and during CI pipelines; desync fails loudly.
- **Risk**: `TaskEntry` struct is used in prompts/logging; removing fields could break callers.
  **Mitigation**: Grep audit of `Domain`/`Scope` usage during implementation; remove all call sites atomically.

## Implementation Notes

- Update `model.TaskFileMeta` and `model.TaskEntry` in `internal/core/model/model.go`.
- Update `prompt.ParseTaskFile` in `internal/core/prompt/common.go:51-73` to populate `Title` from frontmatter and validate non-empty.
- Update `skills/cy-create-tasks/references/task-template.md` to show the new frontmatter shape.
- Update `skills/cy-create-tasks/references/task-context-schema.md` to document field semantics.
- The H1 parser in migration needs to handle `# Task N: <title>`, `# Task N - <title>`, and `# <title>` variants (strip a leading `Task N:` or `Task N -` prefix if present).
