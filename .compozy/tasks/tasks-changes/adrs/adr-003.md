# ADR-003: Validation Command Architecture â€” `compozy validate-tasks` + `start` Preflight

## Status

Accepted

## Date

2026-04-04

## Context

With the new typed `type` field (ADR-002) and required `title` (ADR-001), we need a mechanism to ensure tasks are well-formed **before** they reach execution. Three moments matter:

1. **Immediately after `cy-create-tasks` generates files** â€” the LLM may have picked invalid types or desynced titles.
2. **Before `compozy start` executes tasks** â€” the user may have hand-edited files since generation.
3. **In CI pipelines** â€” detect drift on PRs.

A single validation entry point that can be invoked from all three paths is simpler than scattering checks. The `cy-create-tasks` skill should enforce validation as a post-generation step, and `compozy start` should block execution when validation fails, giving the user a chance to fix before running any agents.

## Decision

Introduce **`compozy validate-tasks`** as a standalone CLI command, and embed the same validator as a preflight check inside `compozy start`.

### Command: `compozy validate-tasks`

- **Usage**: `compozy validate-tasks [--name <feature>] [--tasks-dir <path>] [--format text|json]`
- **Scope**: Validates every `task_*.md` file under the target directory.
- **Checks** (all fields must pass):
  1. `title` is present and non-empty.
  2. Frontmatter `title` matches body H1 (ignoring `Task N:` / `Task N -` prefix).
  3. `type` is present and is a member of the resolved `TypeRegistry`.
  4. `status` is one of `pending|in_progress|completed|blocked`.
  5. `complexity` is one of `low|medium|high|critical` (or empty).
  6. Every value in `dependencies` refers to an existing task file in the same directory.
  7. Frontmatter does **not** contain legacy fields `scope` or `domain` (these yield a **deprecation error**, not a warning â€” they must be migrated out before starting).
- **Exit codes**: `0` on all valid, `1` on any validation failure, `2` on I/O or config errors.
- **Output**: Human-readable by default; `--format json` emits structured results for CI parsing.
- **LLM fix prompt**: When validation fails, the command prints a **ready-to-paste prompt** listing offending files + issues + the allowed types, suitable for feeding into an LLM to auto-correct.

### Preflight in `compozy start`

When `compozy start` resolves tasks, it runs the same validator **before** job setup. If validation fails:

- **TTY mode**: Present a Bubble Tea form blocking execution. The form lists the offending tasks + issues, and offers three actions:
  - **Continue anyway** (acknowledge, proceed â€” for emergencies)
  - **Abort** (exit 1, no jobs created)
  - **Copy fix prompt** (writes the LLM fix prompt to stderr or clipboard helper, then exits)
- **Non-TTY mode**: Print the issues + fix prompt to stderr, exit 1 unless `--force` flag is passed.

A new CLI flag `--skip-validation` (not the same as `--force`) exists for CI pipelines that pre-validate separately.

### Skill enforcement

The `cy-create-tasks` SKILL.md gains a new final step: **"Run `compozy validate-tasks --name <feature>` and do not mark the skill complete until it exits 0. If validation fails, fix the reported issues and re-run."**

## Alternatives Considered

### Alternative 1: Validation only inside `compozy start` (no standalone command)

- **Description**: Bundle validation into start, no separate CLI.
- **Pros**: Less surface area.
- **Cons**: Can't validate from the skill without invoking start (too heavy); no CI story.
- **Why rejected**: Separation of concerns matters. Validation is a fast, read-only operation that deserves its own command.

### Alternative 2: Warning-only in start (never block)

- **Description**: Print warnings, always continue.
- **Pros**: Simplest UX.
- **Cons**: Defeats the purpose of validation; users will ignore warnings and run bad metadata.
- **Why rejected**: User explicitly requested a blocking form.

### Alternative 3: No interactive form in start â€” always require --force

- **Description**: If invalid, exit 1 with fix prompt printed; user reruns with `--force` or fixes.
- **Pros**: Deterministic; works identically in TTY and non-TTY.
- **Cons**: Loses the interactive "copy fix prompt" affordance the user requested.
- **Why rejected**: The interactive TUI form is explicitly in-scope per the product ask.

### Alternative 4: Validate only at file-write time in the skill (no runtime check)

- **Description**: Skill validates right after writing; no start-time check.
- **Pros**: Single validation point.
- **Cons**: Hand-edits between generation and start go uncaught.
- **Why rejected**: The user wants a safety net at start time.

## Consequences

### Positive

- One canonical validator, three call sites (CLI, skill, start) â€” DRY.
- Start-time block prevents wasted agent runs on bad metadata.
- LLM fix prompt turns a failure into a one-paste recovery.
- CI integration is trivial (`compozy validate-tasks` + exit code).

### Negative

- Adds a new top-level CLI command to learn.
- Adds a preflight step that can surprise users on first upgrade; mitigated by the interactive form clearly explaining the issue.
- Bubble Tea form must handle keyboard input, which adds TUI complexity.

### Risks

- **Risk**: Users hit the form, pick "Continue anyway", and never fix the metadata.
  **Mitigation**: The form logs the decision and the warning persists on every start until fixed.
- **Risk**: Non-TTY environments (CI, IDE extensions) can't show the form and get stuck.
  **Mitigation**: TTY detection + `--force`/`--skip-validation` flags; clear error message.

## Implementation Notes

- New command lives in `internal/cli/validate_tasks.go` and calls into `internal/core/tasks/validate.go`.
- Validator is a pure function: `Validate(ctx, tasksDir, typeRegistry) (Report, error)`.
- The Bubble Tea form is a new component in `internal/core/run/` (keep it a standalone modal, not a full view), reuses `lipgloss` styles from `ui_styles.go`.
- The fix-prompt builder is a small string-template function â€” keep it under 40 lines.
