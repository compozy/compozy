# ADR-002: Task Type Taxonomy ‚Äî Constrained Enum with User Override

## Status

Accepted

## Date

2026-04-04

## Context

The `type` field in task metadata is currently free-text. LLMs generating tasks via the `cy-create-tasks` skill produce values like `"Feature Implementation"`, `"Bug Fix"`, `"Refactor"` ‚Äî no two runs produce identical taxonomies. This prevents:

- Filtering/grouping tasks by type in the TUI or reports.
- Routing tasks to specialized providers/skills based on type.
- Consistent historical analysis across projects.

The product direction is to make `type` semantically meaningful: a constrained vocabulary that downstream tools can rely on. Different teams have different taxonomies (a frontend-only team may want `ui`, `state`, `styling`; an API team may want `endpoint`, `migration`, `worker`), so the list must be **user-configurable**.

## Decision

Define task `type` as a **constrained enum validated against a configurable allowlist**.

**Built-in default list** (8 values): `frontend`, `backend`, `docs`, `test`, `infra`, `refactor`, `chore`, `bugfix`.

**User override** via `.compozy/config.toml`:

```toml
[tasks]
types = ["frontend", "backend", "docs", "test", "mobile", "api"]
```

**Semantics**:
- If `[tasks].types` is **present and non-empty**, it **replaces** the built-in defaults entirely.
- If `[tasks].types` is **absent**, the built-in defaults apply.
- Values are lowercase kebab-or-snake identifiers (regex `^[a-z][a-z0-9_-]{1,31}$`); validated at config load.
- Duplicates rejected at load.
- Empty lists rejected at load (use omission to keep defaults).

**Resolution point**: A single `tasks.TypeRegistry` is built from workspace config at CLI startup and passed down to validators, the skill prompt builders, and `compozy start` preflight. No per-command recomputation.

The `cy-create-tasks` SKILL.md instructs the LLM to: (1) read `.compozy/config.toml` and extract `[tasks].types`, (2) fall back to the documented built-in defaults if absent, (3) emit only values from the resolved list, (4) run `compozy validate-tasks` after generation.

## Alternatives Considered

### Alternative 1: Merge list (user adds to defaults)

- **Description**: User config adds `extra_types` that appends to built-ins.
- **Pros**: Convenient; no need to re-list built-ins.
- **Cons**: Hides which values come from user vs. Compozy; user can't drop `chore` or `bugfix` if they don't want them.
- **Why rejected**: Explicit is better than implicit. Users who want to extend can paste the built-ins and add.

### Alternative 2: Keyed map with metadata (labels, colors, icons)

- **Description**: `[tasks.types.frontend] label = "Frontend" color = "#00ff00"`.
- **Pros**: Enables rich TUI rendering per-type.
- **Cons**: More verbose config; solves a styling problem we don't yet have. The header badge (ADR-001) only needs the slug.
- **Why rejected**: YAGNI. Promote to this shape later if styling per-type becomes needed.

### Alternative 3: Leave `type` as free-text, validate only via convention

- **Description**: No enum; skill recommends values but doesn't enforce.
- **Pros**: Zero schema rigidity.
- **Cons**: Doesn't solve the original problem ‚Äî LLMs will continue to drift.
- **Why rejected**: The whole point is to constrain the taxonomy.

## Consequences

### Positive

- Deterministic taxonomy across runs and projects.
- `validate-tasks` has a concrete pass/fail check.
- Teams can tune the taxonomy without forking Compozy.
- TUI badge rendering has a stable input set.

### Negative

- LLM must read the config file before emitting task types; adds one step to the skill workflow.
- Projects that previously used `type: "Feature Implementation"` now fail validation until migrated.

### Risks

- **Risk**: Users set obscure custom types, LLMs struggle to classify correctly.
  **Mitigation**: Skill instructions require the LLM to map task intent to the **closest** allowed type; the default list covers common cases.
- **Risk**: Two projects with different type lists become hard to compare across reports.
  **Mitigation**: Accepted trade-off ‚Äî per-project customization is a feature, not a bug.

## Implementation Notes

- New file: `internal/core/tasks/types.go` ‚Äî exports `TypeRegistry`, `DefaultTypes()`, `NewRegistry(configured []string) (*TypeRegistry, error)`, `(*TypeRegistry).IsAllowed(string) bool`, `(*TypeRegistry).Values() []string`.
- Extend `workspace.ProjectConfig` with `Tasks TasksConfig` field and `[tasks]` TOML section.
- Validator called from `workspace.Validate()` on config load.
- The `cy-create-tasks` SKILL.md gains a new first step: "Read `.compozy/config.toml` ‚Äî if `[tasks].types` exists use it; otherwise use defaults `[frontend, backend, docs, test, infra, refactor, chore, bugfix]`."
