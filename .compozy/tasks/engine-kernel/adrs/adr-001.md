# ADR-001: Service Kernel Pattern with Typed Per-Command Handlers

## Status

Accepted

## Date

2026-04-05

## Context

Compozy today exposes its engine through a Go facade in `internal/core/api.go` with six exported functions: `Prepare`, `Run`, `FetchReviews`, `Migrate`, `Sync`, `Archive`. Each CLI command in `internal/cli/` calls one of these directly via a `runWorkflow func(context.Context, core.Config) error` injection seam in `commandState`. The `core.Config` struct carries ~30 fields covering every mode (PRD tasks, PR reviews, ad-hoc exec), every flag, and every runtime option.

Upcoming phases will add a second ingress: a daemon exposing JSON-RPC that invokes the same engine operations. If we build that ingress by having it call `core.Run()` and siblings directly, we create two parallel call sites for every engine operation. Every new mode, every validation change, every result field must be implemented, tested, and kept equivalent across both paths. That contract is the most expensive thing to maintain in software.

A second problem: the monolithic `core.Config` struct mixes concerns. `run.start` wants completely different fields than `workflow.sync` or `runtime.list`, but today they all pass the same 30-field struct and the handler internally reads only a subset. External consumers of the future SDK/RPC cannot discover which fields are meaningful for which operation without reading implementation code.

The council review of an earlier design, plus the Codex architecture review, both converged on the same structural move: unify execution on a service kernel layer with typed command handlers, then build multiple ingresses (Cobra, RPC) as thin adapters over that kernel.

## Decision

Introduce a new package `internal/core/kernel/` that owns a command dispatcher and typed per-command handlers. Every operation currently exposed in `core/api.go` becomes a typed command with its own input struct, output struct, and handler. The CLI's `runWorkflow` injection point receives a dispatcher instance and routes to commands instead of calling `core.Run()` directly. Phase B's RPC server will use the same dispatcher.

Command dispatch uses Go 1.23 generics for type safety:

```go
type Handler[C any, R any] interface {
    Handle(ctx context.Context, cmd C) (R, error)
}

type Dispatcher struct {
    mu       sync.RWMutex
    handlers map[reflect.Type]any
}

func Register[C any, R any](d *Dispatcher, h Handler[C, R]) { ... }
func Dispatch[C any, R any](ctx context.Context, d *Dispatcher, cmd C) (R, error) { ... }
```

Each command is a typed struct:

```go
type RunStartCommand struct {
    WorkspaceRoot string
    Name          string
    Mode          model.ExecutionMode
    IDE           string
    Model         string
    // ...only fields relevant to starting a run
}

type RunStartResult struct {
    RunID       string
    ArtifactsDir string
    // ...only fields returned by run-start
}
```

The dispatcher lives in the process. The Cobra CLI instantiates it at startup and registers all built-in handlers. Each Cobra command translates its flags into the appropriate command struct, calls `kernel.Dispatch`, and renders the result. Phase B's RPC server will use the same dispatcher as the backing layer behind JSON-RPC methods.

The **Phase A command set** mirrors exactly what `core/api.go` exports today, split by concern. No new user-facing operations are introduced in this refactor:

- `RunStart` — wraps current `core.Run()` (batch modes: PRDTasks, PRReview, Exec)
- `WorkflowPrepare` — wraps current `core.Prepare()`
- `WorkflowSync` — wraps current `core.Sync()`
- `WorkflowArchive` — wraps current `core.Archive()`
- `WorkspaceMigrate` — wraps current `core.Migrate()`
- `ReviewsFetch` — wraps current `core.FetchReviews()`

That is six commands, one per exported function in `internal/core/api.go` today, plus we split the `exec` mode of `core.Run()` into its existing subpath. Total Phase A handlers: 6.

**Deferred commands** (NOT in Phase A scope; documented here to reserve command naming for Phase B):

- Phase B (daemon + RPC): `RunCancel`, `RunStatus`, `RunList`, `RunGet`, `RunSubscribe`, `RuntimeList`, `RuntimeCheck`, `WorkflowValidate`
- Phase D (SDKs + OpenRPC): whatever Phase B needs plus any introspection commands

Deferred commands do not have handlers in Phase A. They are reserved names to prevent future churn.

## Alternatives Considered

### Alternative 1: Keep `core.Run()` and siblings as the seam

- **Description**: Let the RPC server call `core.Run()`, `core.FetchReviews()`, etc. directly. No kernel layer.
- **Pros**: Zero refactor cost. Existing facade works.
- **Cons**: Two ingresses share no layer; every change requires equivalence work on both paths. `core.Config` monolith remains as the public contract, exposing all 30 fields as one surface.
- **Why rejected**: This is the two-path problem Codex explicitly warned against. Ship once, pay forever.

### Alternative 2: Single `Command` interface with a type union

- **Description**: One interface `type Command interface { Kind() string }` where each implementation carries its own payload. Dispatcher unwraps via type switch.
- **Pros**: Simpler registry (no reflection). Matches proto-oneof pattern.
- **Cons**: Loses compile-time type safety at the call site. `Dispatch` returns `any` and the caller must assert. Bug-prone.
- **Why rejected**: Go 1.23 generics eliminate the need for this trade-off. Typed handlers are strictly better.

### Alternative 3: Mediator-pattern framework (`go-mediatr`)

- **Description**: Use a community CQRS/mediator library with pipeline middleware, validation hooks, etc.
- **Pros**: Pre-built middleware chain, pipelines.
- **Cons**: Adds a dependency for a pattern that is ~80 lines of idiomatic generic Go. Middleware abstractions make stack traces harder to read. Overkill for 15 commands.
- **Why rejected**: The custom dispatcher costs less than learning the library.

## Consequences

### Positive

- One execution layer. CLI and future RPC both call the same kernel, guaranteeing behavioral parity by construction.
- Each command has a typed input/output — self-documenting, generator-friendly (OpenRPC in Phase D can be produced mechanically), and minimal for each operation.
- Handlers become unit-testable in isolation without CLI or RPC scaffolding.
- `core.Config` monolith is retired in favor of per-command structs. External consumers only see the fields that matter for their call.

### Negative

- Refactor touches the CLI wiring and the core facade. Tests assert against the old `core.Config` shape need updating.
- Reflection is used once for type-keyed registry lookup (unavoidable for heterogeneous handler storage).
- Every new operation now requires defining an input struct, output struct, and handler — slightly more ceremony than adding a method to `core/api.go`.

### Risks

- **Drift between Cobra flags and command structs**: when a new flag is added, both the Cobra command and the kernel command struct must be updated. Mitigation: the Cobra command is the only caller that maps flags → command struct; drift is local and detectable in CLI tests.
- **Handler registration order**: if a handler is forgotten at startup, `Dispatch` fails at runtime rather than compile time. Mitigation: a startup self-test registers a canary and asserts all expected command types are present.

## Implementation Notes

**Package path**: `internal/core/kernel/` (dispatcher + registry), `internal/core/kernel/commands/` (typed command/result structs per domain).

**Handler dependencies** — every handler needs the same infrastructure, grouped into `KernelDeps`:

```go
type KernelDeps struct {
    Logger      *slog.Logger
    EventBus    *events.Bus[events.Event]
    Workspace   workspace.Context       // loaded at CLI startup
    AgentRegistry agent.Registry        // from internal/core/agent
}
```

`KernelDeps` is constructed once at CLI startup in `internal/cli/root.go` and passed to `kernel.BuildDefault(deps KernelDeps) *Dispatcher`, which instantiates and registers all six Phase A handlers.

**CLI injection path**: the existing `commandState.runWorkflow func(context.Context, core.Config) error` signature is PRESERVED. The dispatcher is captured via closure at command construction time:

```go
// In internal/cli/root.go, at startup:
deps := kernel.KernelDeps{Logger: logger, EventBus: bus, Workspace: ws, AgentRegistry: reg}
dispatcher := kernel.BuildDefault(deps)

// Each command's constructor wraps dispatch in the existing runWorkflow signature:
func newStartCommand(dispatcher *kernel.Dispatcher) *cobra.Command {
    state := newCommandState(kindStart, core.ModePRDTasks)
    state.runWorkflow = func(ctx context.Context, cfg core.Config) error {
        cmd := commands.RunStartFromConfig(cfg)  // typed translation
        _, err := kernel.Dispatch[commands.RunStartCommand, commands.RunStartResult](ctx, dispatcher, cmd)
        return err
    }
    // ...wire flags...
    return cobraCmd
}
```

This keeps the `commandState` contract stable (existing tests continue to inject mock `runWorkflow` closures), while migrating the actual execution layer behind the curtain.

**Transitional shape**: during the refactor, the existing `core.Config` struct stays as a transitional shape. Each `commands.*FromConfig` function maps fields from the legacy `core.Config` to the per-command struct. Once all six handlers are in place, `core.Config` exports are removed and callers build command structs directly.

**Handler implementations call existing engine code unchanged**: `RunStart.Handle` calls `plan.Prepare` then `run.Execute`. `WorkflowSync.Handle` calls the existing `syncTaskMetadata`. No core execution logic is rewritten; only the call sites move behind the dispatcher.
