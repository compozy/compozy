# ADR-003: Event Taxonomy with Schema Versioning and Complete Side-Effect Coverage

## Status

Accepted

## Date

2026-04-05

## Context

Once events become the public contract between the engine and its consumers (journal, TUI, future RPC subscribers, future hooks, future SDKs), the taxonomy of those events is load-bearing. External consumers will read, store, and route on the event kinds we define. Changing them is expensive. Getting them incomplete means downstream consumers cannot faithfully reconstruct run history.

Codex's architecture review identified a critical gap in the initial draft taxonomy: the engine performs post-execution mutations in `execution.go:afterTaskJobSuccess()` and `execution.go:afterReviewJobSuccess()` that are not currently emitted as events. These functions:

- Mutate task markdown files (YAML frontmatter status changes)
- Refresh `.compozy/tasks/<name>/.meta.toml` aggregated metadata
- Mutate review issue files and round metadata
- **Call external provider APIs** (GitHub, CodeRabbit) to post resolved-comment acknowledgments

If `events.jsonl` is the source of truth for what happened during a run, and these actions are not represented as events, the log is incomplete. Consumers replaying a run cannot see "the task file was updated," "the round metadata was refreshed," or "a GitHub comment was posted" — the most consequential side effects of the engine are invisible to replay.

Additionally, the engine will continue to evolve. We need forward-compatibility: older consumers must tolerate events they don't understand, and newer consumers must distinguish event schema versions.

## Decision

Define a complete event taxonomy with 27 event kinds organized by 9 domains. Every event envelope carries `schema_version`, `run_id`, `seq`, `timestamp`, `kind`, and `payload`. All post-execution mutations and external provider calls are modeled as first-class events.

**Event envelope:**

```go
type Event struct {
    SchemaVersion string          `json:"schema_version"`  // "1.0"
    RunID         string          `json:"run_id"`
    Seq           uint64          `json:"seq"`             // monotonic per run, assigned by journal
    Timestamp     time.Time       `json:"ts"`
    Kind          EventKind       `json:"kind"`
    Payload       json.RawMessage `json:"payload"`
}
```

**Event taxonomy (v1):**

| Domain | Event Kinds |
|--------|-------------|
| Run lifecycle | `run.queued`, `run.started`, `run.completed`, `run.failed`, `run.cancelled` |
| Job lifecycle | `job.queued`, `job.started`, `job.attempt_started`, `job.attempt_finished`, `job.retry_scheduled`, `job.completed`, `job.failed`, `job.cancelled` |
| ACP session | `session.started`, `session.update` (wraps `model.SessionUpdate`), `session.completed`, `session.failed` |
| Tool calls | `tool_call.started`, `tool_call.updated`, `tool_call.failed` |
| Usage | `usage.updated` (per-job delta), `usage.aggregated` (run total) |
| Task mutations (post-execution) | `task.file_updated`, `task.metadata_refreshed` |
| Review mutations (post-execution) | `review.status_finalized`, `review.round_refreshed`, `review.issue_resolved` |
| Provider I/O | `provider.call_started`, `provider.call_completed`, `provider.call_failed` |
| Shutdown | `shutdown.requested`, `shutdown.draining`, `shutdown.terminated` |

**Payload contracts** are defined per kind as Go structs in `pkg/compozy/events/kinds/` (public). Each struct has its own JSON tag schema, documented in `docs/events.md`.

**Schema versioning rules:**

- `schema_version` is `"<major>.<minor>"` (e.g., `"1.0"`)
- Additive fields within a minor version: safe, consumers must tolerate unknown fields
- Field rename, removal, or semantic change: bumps major version (`"2.0"`)
- A major bump retires older consumers; migration guides in docs
- The journal records every event's schema version; replay consumers can route by version

**Post-execution mutation events** carry enough metadata for consumers to reconstruct the action without reading the filesystem:

```go
type TaskFileUpdatedPayload struct {
    TasksDir    string `json:"tasks_dir"`
    TaskName    string `json:"task_name"`
    FilePath    string `json:"file_path"`
    OldStatus   string `json:"old_status"`
    NewStatus   string `json:"new_status"`
}

type ReviewIssueResolvedPayload struct {
    ReviewsDir   string `json:"reviews_dir"`
    IssueID      string `json:"issue_id"`
    Provider     string `json:"provider"`
    PR           string `json:"pr"`
    ProviderRef  string `json:"provider_ref"`
    PostedAt     time.Time `json:"posted_at"`
}

type ProviderCallCompletedPayload struct {
    Provider     string        `json:"provider"`
    Endpoint     string        `json:"endpoint"`
    Method       string        `json:"method"`
    StatusCode   int           `json:"status_code"`
    DurationMs   int64         `json:"duration_ms"`
    PayloadBytes int           `json:"payload_bytes"`
}
```

## Mutation/Event Ordering Policy

Each mutation-producing call site has one of three emission policies, chosen by the nature of the mutation:

**Policy 1 — emit-after-success** (default for filesystem mutations):
Event emits AFTER the mutation succeeds. If the mutation fails, no event fires; the error propagates. Consumers see only successful mutations in the event log.
Applies to: `task.file_updated`, `task.metadata_refreshed`, `review.status_finalized`, `review.round_refreshed`.

**Policy 2 — emit-on-attempt + emit-on-completion** (for external I/O):
Emit `provider.call_started` BEFORE the call, then `provider.call_completed` OR `provider.call_failed` AFTER. Both events share the same `call_id` field for correlation. Consumers can reconstruct full lifecycle including in-flight calls if a crash occurs between the two events.
Applies to: `provider.call_started`/`provider.call_completed`/`provider.call_failed`, `review.issue_resolved` (compound: emits `provider.call_started` → provider API call → `review.issue_resolved` if success, `provider.call_failed` if not).

**Policy 3 — emit-on-decision** (for flow-control events):
Emit when a decision is made, regardless of what happens next. `run.queued` emits when the run is enqueued; `run.started` emits when execution begins; `run.cancelled` emits when cancellation is requested.
Applies to: `run.*`, `job.*`, `shutdown.*`.

**No compensating events in v1**: if a mutation succeeds and its event is emitted, but a subsequent step in the handler fails, we do NOT emit a "rollback" or "compensate" event. The partial state is visible in the event log as-is. Consumers replaying the log see the same partial state a second run would produce.

## Provider Call Failure Policy

Provider API calls (GitHub comment posting, CodeRabbit acknowledgments) are WARN-AND-CONTINUE in v1:

- Call failure emits `provider.call_failed` event with status code and error message
- The parent operation (e.g., `afterReviewJobSuccess`) logs a warning, emits the `review.issue_resolved` event WITH a `provider_posted=false` field, and continues
- The run is NOT failed by a provider call failure
- The run's final status is unaffected by provider outcomes

Rationale: provider calls are side effects that acknowledge work already done. Failing a completed run because GitHub was unreachable would punish the user for external infrastructure. The event log records the failure; operators can reconcile manually or retry later via a Phase B tool.

This policy is explicit to prevent accidental regression: if future code treats provider call failure as run-fatal, that is a breaking contract change requiring its own ADR.

## Removed Event Kinds

Three event kinds from the original draft were removed after a code-site audit:

- **`prompt.built` and `prompt.written`** (entire Prompt domain removed): No emission site exists in the codebase. Prompt generation in `plan/prepare.go` and file writes in `exec_flow.go` are silent I/O with no consumer that needs notification. These would be artificial telemetry events emitted solely for completeness, with no replay or observability value. If a future phase needs prompt lifecycle events, they can be added as a minor version bump.

- **`tool_call.completed`**: Redundant with `tool_call.updated` carrying `ToolCallStateCompleted`. The ACP session implementation (`agent/session.go`) does not distinguish between a final update and a completion — both flow through `UpdateKindToolCallUpdated`. Adding a separate completion event would require splitting a single code path into two for no consumer benefit.

This reduces the taxonomy from 34 to 27 event kinds across 9 domains (down from 10).

## Alternatives Considered

### Alternative 1: Ad-hoc event kinds added as needed

- **Description**: Define a minimal set of events now, add more as consumers demand them.
- **Pros**: Ships faster; avoids over-design.
- **Cons**: External consumers cannot rely on the log being complete. Reading events.jsonl and running an equivalent real run produces different filesystem state (mutations unobserved). Every new event kind is a breaking change for consumers that exhaustively match.
- **Why rejected**: Consumers NEED completeness for replay to be meaningful. Incomplete v1 creates incentive for consumers to read files directly, defeating the log as source-of-truth.

### Alternative 2: Protobuf/FlatBuffers schema

- **Description**: Define events in `.proto` files, generate Go structs, use binary encoding in the journal.
- **Pros**: Strong schema evolution story; efficient binary format; cross-language bindings for free.
- **Cons**: Adds protoc to the build toolchain. Binary events are not `tail -f | jq` friendly. Conflicts with NDJSON human-inspectability that aligns with existing artifact layout.
- **Why rejected**: JSON-in-NDJSON is the canonical format for the `.compozy/runs/<id>/events.jsonl` file that already exists. Breaking that to gain binary efficiency trades observability for perf we don't need.

### Alternative 3: No versioning; use additive fields only

- **Description**: Never rename or remove fields; rely on JSON unknown-field tolerance.
- **Pros**: Simplest possible schema evolution rule.
- **Cons**: Over time, payloads accumulate deprecated-but-present fields; consumers have no signal when a field's semantics change.
- **Why rejected**: Semantic changes happen. Having `schema_version` in the envelope is ~10 bytes per event and enables clean migration paths.

## Consequences

### Positive

- `events.jsonl` is a complete, faithful record of every side effect the engine takes. Replay is semantically meaningful.
- Consumers route on versioned kinds; safe to add new kinds without breaking existing readers.
- Provider I/O events make external calls observable — debugging "did we actually post that GitHub comment?" becomes `grep provider.call_completed`.
- Metrics can be derived directly from the event stream (per-provider call rate, success rate, latency percentiles).

### Negative

- More event kinds than the minimum; more maintenance work to keep payload structs in sync with emitter code.
- Schema version field adds ~15 bytes per event. At 10k events/run this is ~150KB overhead per run — acceptable.
- Emitting post-execution events requires instrumenting existing code paths (`afterTaskJobSuccess`, `afterReviewJobSuccess`, `resolveProviderBackedIssues`, etc.) that currently just mutate state silently.

### Risks

- **Schema drift between emitters and consumers**: payload struct changes in emitter code that aren't reflected in `docs/events.md` or consumer libraries. Mitigation: generate `docs/events.md` from Go structs; CI fails if docs are stale.
- **Kind explosion**: incentive to add a new kind for every small variation leads to 100+ kinds over time. Mitigation: taxonomy review required for new kinds; prefer richer payloads on existing kinds over new kinds.
- **Versioning discipline**: contributors bump minor version accidentally for a major change. Mitigation: every schema version bump requires a changelog entry; CI check compares JSON schemas between commits.

## Implementation Notes

- Package path: `pkg/compozy/events/kinds/` (public — consumed by reader library and Phase B daemon)
- One Go file per domain: `run.go`, `job.go`, `session.go`, `tool_call.go`, `usage.go`, `task.go`, `review.go`, `provider.go`, `shutdown.go`
- `EventKind` is a string enum with const values; each kind has a constructor helper that returns a typed payload
- `docs/events.md` is the canonical public reference; regenerated from code
- Schema version is set per-build from a single constant `events.SchemaVersion = "1.0"` — changes are deliberate single-line edits
- Every new event kind added in a PR must: (1) have a payload struct, (2) have a doc entry, (3) be emitted at an appropriate call site, (4) have a unit test asserting the emitter fires
