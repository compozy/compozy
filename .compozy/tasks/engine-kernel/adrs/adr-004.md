# ADR-004: Journal Upstream of Fanout with Single-Writer Per-Run Model

## Status

Accepted

## Date

2026-04-05

## Context

The engine emits events that must be (a) persisted durably to `.compozy/runs/<run-id>/events.jsonl` for replay and debugging, and (b) fanned out live to subscribers (TUI, Phase B RPC, Phase C hooks). The question is where the persistence sits in the data flow.

The initial design proposal treated the NDJSON writer as just another subscriber on the event bus, symmetric with the TUI. Codex's architecture review flagged this as wrong: if the writer is a subscriber, a crash between publish and file append loses the event forever, and live subscribers see events the journal has not yet recorded. Concretely: the current executor already has two independent lossy paths in `logging.go:HandleUpdate()` that drop snapshot and usage messages under load; symmetric-subscriber persistence would layer a third lossy path.

The correct architectural move is: events flow into a journal that (a) assigns a monotonic sequence number per-run, (b) appends the envelope to `events.jsonl`, and (c) THEN publishes to the bus for fanout. Subscribers downstream of the journal receive events that are already durable. Replay readers and live subscribers see the same sequence of events with the same seq numbers.

This is the architectural shape used by most event-sourced systems (Kafka's log, PostgreSQL's WAL, SQLite's WAL): the log is the canonical order; consumers derive views from it. Compozy's `events.jsonl` is the log.

Additionally, the journal must handle concurrency. The engine runs jobs concurrently within a single run (worker pool from `cfg.Concurrent`). Each worker may emit events simultaneously. The journal writer must serialize writes so that (a) seq numbers are monotonically increasing with no gaps, (b) file appends are atomic (one complete JSON object per line, never interleaved bytes), (c) fsync has meaningful durability semantics.

## Decision

The journal is a single-writer goroutine per-run that owns the `events.jsonl` file. Event emitters submit events to the journal via a buffered channel; the writer goroutine assigns seq, encodes to JSON, appends to the file, and publishes to the bus. Persistence happens BEFORE bus publish.

**Data flow:**

```
  Event emitters                    Journal goroutine                       Bus subscribers
 (executor, ACP handler,          (single-writer per-run)                (TUI, Phase B RPC, hooks)
  post-exec mutations)                      │
        │                                   │
        │  journal.Submit(ev) ──► chan Event (bounded cap)
        │                                   │
        │                                   ▼
        │                         1. assign seq
        │                         2. encode + append to events.jsonl
        │                         3. batch-flush if threshold hit
        │                         4. bus.Publish(enrichedEvent)  ──► subscribers
```

**Writer model:**

- One journal instance per run, created during `plan.Prepare` (which allocates run artifacts)
- `Submit(ev Event) error` is safe for concurrent callers; under the hood it sends to a bounded channel (default capacity 1024)
- If the channel is full, `Submit` blocks with a context deadline (same pattern as ADR-006's ACP buffer strategy)
- The writer goroutine loops: read event → assign seq → `bufio.Writer.Encode(ev)` → if batch-threshold reached, `Flush` + `f.Sync`
- Batch flush thresholds: every 32 events OR every 100ms, whichever first
- Terminal events (`run.completed`, `run.failed`, `run.cancelled`) force immediate `Flush` + `f.Sync`
- Writer shuts down via context cancellation: drains remaining queued events, final flush + sync, closes file

**File format:**

- Append-only, one JSON object per line, UTF-8, newline-terminated
- Path: `.compozy/runs/<run-id>/events.jsonl` (filename matches what exec mode already writes today; see explore map)
- Writer opens with `os.O_APPEND | os.O_CREATE | os.O_WRONLY`, mode 0644
- `bufio.Writer` with 16KB buffer sits between `json.Encoder` and `*os.File`

**Seq assignment:**

- Monotonic per-run, starts at 1
- Assigned in the writer goroutine ONLY — a plain `uint64` local variable inside the loop
- No `atomic` primitives needed (only the writer reads/writes `seq`)
- Gaps are impossible by construction

**Crash recovery:**

- On daemon/CLI crash, the journal may have buffered events not yet fsynced; those are lost
- Batch flush bounds the loss window: at 100ms batch interval, at most 100ms of events
- `run.json` metadata file is written atomically at run-start and updated on run-terminal; readers can detect "incomplete" runs
- Reader library (ADR-005) tolerates truncated last-line JSON (skip if parse fails)

## Alternatives Considered

### Alternative 1: Journal as parallel subscriber

- **Description**: Journal subscribes to the bus like TUI/RPC subscribers. Bus publishes to all simultaneously.
- **Pros**: Symmetric; simpler bus model; each subscriber independent.
- **Cons**: Crash between publish and journal-append loses events. Live subscribers see events the journal hasn't persisted, breaking the log-as-truth property. Subscriber ordering becomes non-deterministic.
- **Why rejected**: Fast-first vs durable-first trade-off lands on durable-first for a dev tool where runs take 20+ minutes — losing the last minute of events is unacceptable.

### Alternative 2: Write-per-event with fsync per event

- **Description**: Append and fsync every single event before fanout.
- **Pros**: Maximum durability — zero lost events.
- **Cons**: fsync per event is expensive (~1-10ms on typical SSDs). At 100 events/sec sustained, this dominates the event loop.
- **Why rejected**: Wrong perf trade-off. Batch flush with 100ms interval loses at most 100ms of events on crash — acceptable for this tool.

### Alternative 3: Event persistence inside each subscriber

- **Description**: Subscribers that care about durability (a future "persistent-subscriber" RPC client) manage their own local cursors.
- **Pros**: Separates concerns.
- **Cons**: No canonical ordering; every subscriber sees its own stream; replay is a cross-subscriber merge nightmare. Essentially reinvents Kafka consumer groups in a single-binary CLI.
- **Why rejected**: Architectural complexity vastly exceeds the benefit for a local-first tool.

### Alternative 4: SQLite WAL for journal

- **Description**: Use SQLite's built-in WAL for event storage.
- **Pros**: Transactional guarantees, native replay queries.
- **Cons**: Deferred to Phase B per ADR scope. Adds a dependency and a schema to manage for what is fundamentally an append-only log. NDJSON files compose better with `.compozy/runs/<id>/` layout and are human-inspectable.
- **Why rejected**: Deferred to Phase B catalog (separate concern). NDJSON is the right format for per-run event streams.

## Consequences

### Positive

- `events.jsonl` is the canonical ordered log of a run; all live subscribers observe a subset of what is already persisted.
- Replay is a simple file read; seq numbers guarantee ordering.
- Crash loss window is bounded and documented (max 100ms at default batch interval).
- Provider I/O events (ADR-003) are durably recorded — you can investigate "did we post that GitHub comment?" by reading the file.

### Negative

- Events experience a small latency increase (batch flush interval) before reaching live subscribers.
- The journal becomes a throughput bottleneck for the run. Mitigation: sized buffer + batch flush keeps this below the event emission rate even at high ACP update rates.
- One journal goroutine per run = one more goroutine per concurrent run. Trivial.

### Risks

- **Buffer overflow under bursty load**: 1024-event buffer can fill during intense ACP streaming. Mitigation: backpressure with 5s deadline (same pattern as ADR-006). If deadline exceeded, log explicit drop + metric counter.
- **Partial writes on crash**: the last buffered batch may be half-written. Mitigation: readers tolerate parse-fail on final line.
- **Multi-process writers**: if two processes somehow write to the same run's journal, file corruption is possible. Mitigation: kernel owns the file for the run's lifetime; path names include run-id making concurrent writes impossible.

## Implementation Notes

- Package path: `internal/core/run/journal/`
- Key types:
  - `Journal` — per-run writer
  - `JournalConfig` — batch thresholds, buffer capacity
  - `Submit(ctx context.Context, ev Event) error` — enqueue event, respects ctx timeout
- The journal is constructed in `plan.Prepare` alongside other run artifacts (it needs `RunArtifacts.EventsPath`)
- The journal is passed to the kernel command handler for `RunStart`; the handler passes it to `run.Execute` which passes it to workers
- On `run.Execute` return (success or failure), the journal is `Close()`d with context deadline; Close drains remaining queued events, writes terminal event, final flush + sync
- Metrics exposed: `journal_events_written_total`, `journal_flush_duration_seconds`, `journal_drops_total`, `journal_buffer_depth`
