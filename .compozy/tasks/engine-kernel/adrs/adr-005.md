# ADR-005: Reader Library over `.compozy/runs/` using nxadm/tail and fsnotify

## Status

Accepted

## Date

2026-04-05

## Context

Once the journal (ADR-004) persists every event to `.compozy/runs/<run-id>/events.jsonl`, external consumers need a way to read, replay, tail, and enumerate runs without re-parsing file formats in every consumer. The reader library becomes the first stable public API of Compozy — the substrate on which Phase B's daemon, Phase C's hook system, and Phase D's SDKs all read run state.

Consumers need four capabilities:

1. **List runs**: enumerate all runs in a workspace, with metadata (status, mode, IDE, started_at, ended_at)
2. **Open a run**: load `run.json` metadata and expose artifact paths
3. **Replay**: iterate the full event history of a completed (or in-progress) run, starting from any seq cursor
4. **Tail**: subscribe to live events as they're appended to `events.jsonl`, handling file rotation and truncation edge cases correctly

Building a correct file-tailer from scratch is deceptively difficult. Rotation (file replaced with a new inode), truncation (file shrunk), and partial-line buffering at the write boundary are all edge cases that have been solved repeatedly in the Go ecosystem. Our library research identified `github.com/nxadm/tail` (a maintained fork of the abandoned `hpcloud/tail`) as the Ginkgo-blessed, cross-platform solution.

Directory watching — detecting new runs appearing in `.compozy/runs/` — needs `fsnotify`, which is already a transitive dependency via `nxadm/tail` and is the de-facto Go standard (used by Kubernetes, Prometheus, Vault, Terraform).

## Decision

Create a public Go package `pkg/compozy/runs/` exposing typed access to the on-disk contract. The package depends on two new direct dependencies: `github.com/nxadm/tail` and `github.com/fsnotify/fsnotify`. Both are pure Go with low transitive footprints.

**Package dependency**: This package consumes `pkg/compozy/events/` for event types. The events package is PUBLIC (lives under `pkg/`) because external consumers of the reader library must be able to name the types they receive.

**Public API (v1, frozen contract):**

```go
package runs

import (
    "context"
    "iter"
    "time"
    "github.com/compozy/compozy/pkg/compozy/events"
)

// RunSummary is the public shape of a run's metadata.
type RunSummary struct {
    RunID         string
    Status        string       // "running", "completed", "failed", "cancelled"
    Mode          string       // "prd-tasks", "pr-review", "exec"
    IDE           string
    Model         string
    WorkspaceRoot string
    StartedAt     time.Time
    EndedAt       *time.Time   // nil if not ended
    ArtifactsDir  string       // absolute path to .compozy/runs/<run-id>/
}

// ListOptions filters the result set.
type ListOptions struct {
    Status []string
    Mode   []string
    Since  time.Time
    Until  time.Time
    Limit  int
}

// List enumerates runs under workspaceRoot's .compozy/runs/ directory.
func List(workspaceRoot string, opts ListOptions) ([]RunSummary, error)

// Open loads a run and returns a handle.
func Open(workspaceRoot, runID string) (*Run, error)

// Run is a handle over an on-disk run.
type Run struct { /* exported: Summary RunSummary; unexported: paths, lazy state */ }

// Summary returns the loaded run metadata.
func (r *Run) Summary() RunSummary

// Replay yields events from fromSeq to end-of-file.
// fromSeq=0 starts from the beginning.
func (r *Run) Replay(fromSeq uint64) iter.Seq2[events.Event, error]

// Tail follows events.jsonl. First replays from fromSeq to current EOF,
// then switches to live-tail. Returns events channel and errors channel.
// Cancel ctx to stop tailing. Both channels close when tailing stops.
func (r *Run) Tail(ctx context.Context, fromSeq uint64) (<-chan events.Event, <-chan error)

// RunEvent is emitted by WatchWorkspace as runs are created or their
// status transitions.
type RunEvent struct {
    Kind    RunEventKind     // "created", "status_changed", "removed"
    RunID   string
    Summary *RunSummary      // non-nil for "created" and "status_changed"
}

type RunEventKind string

const (
    RunEventCreated       RunEventKind = "created"
    RunEventStatusChanged RunEventKind = "status_changed"
    RunEventRemoved       RunEventKind = "removed"
)

// WatchWorkspace emits RunEvent notifications.
// Cancel ctx to stop watching.
func WatchWorkspace(ctx context.Context, workspaceRoot string) (<-chan RunEvent, <-chan error)
```

The `Tail` and `WatchWorkspace` functions return `(<-chan T, <-chan error)` — NOT a close function. Callers control lifetime by cancelling `ctx`. Channels close automatically when `ctx` cancels or a terminal condition is reached.

**Implementation approach:**

- `List` scans the `.compozy/runs/` directory using `os.ReadDir`, reads each `run.json`, filters by options. Cost is O(N) runs per call; acceptable for local dev workflows.
- `Open` loads `run.json` once and holds paths; lazy loading for events.
- `Replay` uses `bufio.Scanner` over `events.jsonl`, skipping lines until `seq >= fromSeq`, yielding typed events. Returns Go 1.23 `iter.Seq2` for ergonomic range-over-func usage.
- `Tail` wraps `nxadm/tail` to follow `events.jsonl`. It first replays from `fromSeq` to the current end-of-file, then switches to live-tail mode. This unified "replay then live" pattern ensures consumers never experience a gap between history and live.
- `WatchWorkspace` uses `fsnotify.Watcher` on the runs directory, emitting events when subdirectories are created (new runs) or when `run.json` is modified (run status transitions).

**Versioning:**

- The package is public (under `pkg/`), meaning its API is subject to semver
- Initial version: `v0.x` (unstable, may break) until Phase B's daemon validates it
- Graduates to `v1.0` when the daemon is in production use

**Consumers of the reader library:**

- `internal/cli/` commands (e.g., `compozy runs list`, `compozy watch <run-id>`)
- Phase B daemon's RPC handlers read state via this library
- Phase D SDKs re-expose this library's types for external users
- Debugging tools, CI integrations, custom scripts

## Alternatives Considered

### Alternative 1: Custom tailer on top of fsnotify + bufio

- **Description**: Write the file-tailer from scratch using `fsnotify` and `bufio.Scanner`, ~100 LOC.
- **Pros**: Zero new library dependencies beyond fsnotify.
- **Cons**: Rotation and truncation edge cases (inode tracking, partial-line buffering across writes) are genuinely hard to get right; nxadm/tail has solved these with years of bug-fixing.
- **Why rejected**: The correctness gain from a proven library outweighs the minimal dependency cost.

### Alternative 2: Expose raw file access only

- **Description**: Don't build a reader library; document the on-disk contract and let consumers read files directly.
- **Pros**: Zero new code; zero API surface to maintain.
- **Cons**: Every consumer re-implements JSON parsing, type mapping, tailing, directory scanning. Bugs proliferate. The on-disk contract becomes harder to evolve because we don't know who reads it and how.
- **Why rejected**: The reader library IS the contract; skipping it pushes complexity to every consumer.

### Alternative 3: Ship runs access as part of the CLI only

- **Description**: Only `internal/cli/` commands can read runs; no public package.
- **Pros**: Smaller public surface.
- **Cons**: Phase B daemon and Phase D SDKs need programmatic access; they would either re-implement the reader internally or be forced to shell out to the CLI. Both are bad.
- **Why rejected**: The library exists to serve non-CLI consumers; making it internal defeats the purpose.

### Alternative 4: `github.com/rjeczalik/notify` for directory watching

- **Description**: Use rjeczalik/notify instead of fsnotify for recursive directory watching.
- **Pros**: Recursive watching built-in on all platforms.
- **Cons**: `.compozy/runs/` is a flat directory (no nesting); recursion is unneeded. Lower community adoption than fsnotify.
- **Why rejected**: fsnotify's feature set matches Compozy's needs exactly; no reason to pay for recursion.

## Consequences

### Positive

- External consumers get a typed, supported API with no file-format knowledge required.
- The on-disk contract is testable through the library: contract tests assert library round-trips match journal writes.
- Phase B daemon is significantly simpler — its run-state endpoints (`run.list`, `run.get`, `run.subscribe`) delegate to this library.
- Debugging tools and custom scripts have a single entry point.

### Negative

- Two new direct dependencies: `nxadm/tail` and `fsnotify`. Both pure Go, MIT/BSD, low transitive footprint (fsnotify depends only on `golang.org/x/sys`).
- Public API surface that must be versioned and supported.
- `nxadm/tail` transitively depends on `go.uber.org/tomb.v1`, which is old. Acceptable for now; library can be replaced by custom tailer later if this becomes a concern.

### Risks

- **API breakage during v0**: library consumers may churn during early Phase B integration. Mitigation: keep it `v0.x` until daemon is stable; communicate breaks via CHANGELOG.
- **Tail lag at high event rates**: `nxadm/tail` polls for new bytes; high-rate streams can experience ~50-200ms delay. Mitigation: acceptable for dev-tool workflows; if needed, expose a `TailPollInterval` option.
- **File handle exhaustion**: long-lived tailers hold open file descriptors. Mitigation: `Tail` returns a close function; callers must defer it.

## Implementation Notes

- Package path: `pkg/compozy/runs/` (public)
- Events types live at `pkg/compozy/events/` (public); reader library imports directly (no re-export needed)
- Add to `go.mod` via `go get github.com/nxadm/tail github.com/fsnotify/fsnotify`
- Tests use `t.TempDir()` + fake `.compozy/runs/` layouts
- `Replay` returns `iter.Seq2[Event, error]` (Go 1.23 range-over-func)
- `Tail` and `WatchWorkspace` return two channels (events, errors); caller stops via `ctx` cancel
- Documentation: `docs/reader-library.md` with usage examples
