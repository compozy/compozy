# ADR-002: Incremental Runtime Supervision Hardening Inside the Existing Daemon Boundary

## Status

Accepted

## Date

2026-04-20

## Context

The daemon runtime already has the right coarse ownership split: `Host` boots the singleton and transports, `RunManager` owns active runs, `global.db` and `run.db` persist state, and startup reconciliation marks interrupted runs as crashed. The analysis pack shows that the main production risks are not caused by the high-level shape itself, but by specific resilience gaps inside it: missing signal handling, unbounded shutdown, no orphan reaping, no process-group termination, no checkpoint on close, and limited recovery honesty when live subprocesses survive a daemon crash.

The approved scope keeps runtime supervision as a hardening track rather than the single structural refactor boundary.

## Decision

Keep the existing `internal/daemon` ownership model and harden supervision incrementally inside that architecture.

The hardening bundle includes:

- signal-driven graceful shutdown for detached daemon processes
- bounded shutdown contexts for HTTP, UDS, DB, and host cleanup
- orphan reaping on stale-owner recovery
- process-group aware subprocess termination
- SQLite checkpoint discipline for `global.db` and `run.db`
- stronger startup and periodic recovery signals for interrupted runs

## Alternatives Considered

### Alternative 1: Make runtime supervision the primary structural refactor

- **Description**: Reshape the daemon around a new runtime ownership boundary before fixing the concrete resilience gaps.
- **Pros**: Could remove some existing coupling more aggressively.
- **Cons**: Higher migration risk, broader blast radius, and slower path to production fixes.
- **Why rejected**: The analysis shows the current high-level runtime split is serviceable; the urgent problems are inside it.

### Alternative 2: Apply only a minimal patch bundle

- **Description**: Fix one or two P0 issues and defer the rest.
- **Pros**: Smallest initial change set.
- **Cons**: Leaves shutdown, orphaning, recovery honesty, and WAL growth partially unresolved.
- **Why rejected**: The daemon is already long-lived enough that partial hardening is not an acceptable finish line.

## Consequences

### Positive

- Faster path to production-grade resilience without rewriting the daemon core
- Lower coupling risk than a deeper runtime redesign
- Clear sequencing: contract cleanup first, runtime hardening immediately after

### Negative

- Existing ownership seams remain and may still need later cleanup
- Some fixes depend on extra runtime metadata, such as child PID or process-group tracking

### Risks

- Process-group behavior differs across Unix and Windows
- Recovery logic can misclassify runs if liveness probes are incomplete or stale

## Implementation Notes

- Keep `Host`, `RunManager`, `globaldb`, and `rundb` as the main ownership surfaces
- Add signal handling and shutdown timeout orchestration at the daemon `Run` layer
- Track enough subprocess identity to distinguish dead runs from orphaned-but-live children
