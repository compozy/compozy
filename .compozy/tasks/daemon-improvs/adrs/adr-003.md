# ADR-003: Validation-First Daemon Hardening

## Status

Accepted

## Date

2026-04-20

## Context

The approved scope includes a canonical transport contract refactor and a runtime supervision hardening bundle. Both touch behavior that can drift silently when only unit tests exist: handler shapes can diverge between HTTP and UDS, stream consumers can mis-handle SSE framing, and ACP subprocess failures can remain untested because current mocks cannot simulate malformed frames or mid-stream disconnects.

The user explicitly chose the highest validation bar for this TechSpec.

## Decision

Make runtime harnesses, HTTP/UDS parity tests, and ACP fault-injection coverage part of the primary implementation scope, not a later follow-up.

This includes:

- a reusable harness that boots a real `compozy` daemon with isolated home and workspace state
- parity coverage across HTTP, UDS, and CLI-facing surfaces where applicable
- an ACP mock driver with fixture-based fault injection for disconnects, invalid frames, and cancellation-sensitive scenarios

## Alternatives Considered

### Alternative 1: Add parity tests only, defer harness and ACP fault injection

- **Description**: Validate transport shape now, but leave deeper runtime validation for later.
- **Pros**: Smaller initial testing investment.
- **Cons**: Leaves the riskiest subprocess and stream failures untested.
- **Why rejected**: It still allows runtime regressions in the highest-risk surfaces.

### Alternative 2: Keep validation mostly unit-test based

- **Description**: Continue with in-process fakes and targeted unit coverage.
- **Pros**: Lowest implementation overhead.
- **Cons**: Does not prove real daemon boot, transport parity, or ACP failure behavior.
- **Why rejected**: The existing gap analysis already shows this is insufficient.

## Consequences

### Positive

- The contract refactor and runtime hardening get regression protection immediately
- Failures become easier to reproduce and debug in CI
- Future daemon work inherits a stable integration-testing foundation

### Negative

- Primary scope gets larger and requires more supporting test infrastructure
- Build and CI strategy must split lanes so integration tests do not bloat the fast unit loop

### Risks

- Harness flakiness can reduce confidence if timeouts and cleanup are not designed carefully
- ACP fixtures can become too broad if they try to model unused AGH surfaces

## Implementation Notes

- Keep the first harness focused on daemon lifecycle, transport parity, and ACP runtime failures
- Use build tags or dedicated targets so `make verify` can stay predictable
- Prefer deterministic fixtures and artifact manifests over ad-hoc env-var scenarios
