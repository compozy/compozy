# ADR-003: Expose Nested Agent Execution Through a Compozy-Owned MCP Server

## Status

Accepted

## Date

2026-04-10

## Context

The user requires reusable agents to be invokable not only from `compozy exec --agent <name>` but also from ACP-backed drivers through a generic tool surface. Earlier concern centered on whether ACP could host custom tool plumbing consistently across runtimes. Protocol research confirmed that ACP sessions already accept `mcpServers` during `session/new`, which provides a standard path for tool exposure.

Current Compozy code does not use this path yet:

- `internal/core/agent/client.go` creates ACP sessions with `McpServers: []acp.McpServer{}`
- `internal/core/run/internal/acpshared/command_io.go` does not carry reusable MCP server configuration through session setup

Compozy must decide whether nested agent execution should be implemented as custom host wiring per runtime or as a shared MCP capability.

## Decision

**Compozy will expose reusable agent execution to ACP-backed runtimes through a single reserved MCP server named `compozy`, implemented as a hidden stdio subcommand, `compozy mcp-serve`, using the official Go SDK `github.com/modelcontextprotocol/go-sdk/mcp`. Compozy will inject that reserved server into ACP session setup for every ACP session it creates, then merge it with any MCP servers declared in the selected agent's `mcp.json` when a reusable agent is in play.**

V1 will expose a generic `run_agent` MCP tool. The tool accepts:

- `name`: resolved agent identifier
- `input`: prompt or task payload for the child agent

The tool returns structured output containing:

- resolved agent metadata
- final assistant text or structured result payload
- execution status and failure details when the child run fails

This MCP server is a Compozy concern, not a property of any one runtime. ACP-backed sessions receive it through `mcpServers`, alongside agent-local MCP servers from `mcp.json`, and nested execution flows back through the normal Compozy agent pipeline.

Child execution semantics are explicit:

- `run_agent` creates a real nested ACP session rather than an inline shortcut
- nested depth is host-owned state, not caller-controlled tool input
- the child session receives a fresh reserved `compozy` MCP server instance
- the child effective access mode is the minimum of the parent effective access mode and the child agent's resolved access mode
- ordinary skill-driven sessions without `--agent` still receive the reserved `compozy` MCP server so bundled skills can invoke `run_agent`

## Alternatives Considered

### Alternative 1: One dedicated tool per agent

- **Description**: Register `run_council`, `run_reviewer`, and similar tools for every discovered agent.
- **Pros**: Friendly tool names inside the runtime.
- **Cons**: Dynamic tool churn, poor scalability, harder collision handling, and more protocol surface to document and test.
- **Why rejected**: The user explicitly asked for a generic surface rather than one tool per agent.

### Alternative 2: Runtime-specific host command registration

- **Description**: Implement custom tool plumbing separately for each ACP-backed runtime adapter.
- **Pros**: Potentially tighter runtime-specific integrations.
- **Cons**: Duplicated logic, uneven behavior across runtimes, and unnecessary divergence from ACP's MCP path.
- **Why rejected**: ACP already provides a standard MCP server hook at session creation time.

### Alternative 3: CLI-only agent execution

- **Description**: Support `compozy exec --agent` but do not support runtime tool invocation in v1.
- **Pros**: Simplest initial release.
- **Cons**: Fails a key requirement and leaves skills like `cy-idea-factory` with only conceptual references to helper agents.
- **Why rejected**: Tool-driven nested execution is part of the feature's intended value.

## Consequences

### Positive

- One cross-runtime integration path for nested agent execution.
- Keeps ACP transport concerns aligned with official protocol capabilities.
- Avoids an explosion of per-agent tools.
- Makes future non-agent MCP tools possible through the same server.

### Negative

- Requires a new internal MCP server implementation and lifecycle management.
- Nested runs add recursion, timeout, and policy concerns that plain `exec` does not have.

### Risks

- **Risk**: Recursive agent chains can create runaway nested execution.
  **Mitigation**: Add depth limits, cycle detection, and explicit nested-run metadata in the MCP request context.
- **Risk**: Different ACP runtimes may vary in their MCP support quality.
  **Mitigation**: Validate against supported runtimes and degrade with a clear capability error when MCP injection is not available.
- **Risk**: Child runs may inherit unsafe or unintended privileges.
  **Mitigation**: Resolve access mode explicitly for nested runs and log the effective policy with the tool result.

## Implementation Notes

- Use a local stdio MCP server launched by Compozy per ACP session via `compozy mcp-serve`.
- Thread merged MCP server definitions through the existing `agent.SessionRequest` and ACP session setup path.
- Attach the reserved server even when no reusable agent was selected for the top-level session.
- Treat host-reserved MCP server names as non-overridable and fail validation on collision with agent-local `mcp.json`.
- Propagate nested depth through host-owned session metadata and internal startup environment, not via the `run_agent` request payload.
- Keep the nested execution path inside Compozy core so `run_agent` reuses the same agent resolution, prompt assembly, and runtime validation as the CLI.
