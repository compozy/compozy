# ADR-005: Use Agent-Local `mcp.json` with Standard MCP Configuration Shape

## Status

Accepted

## Date

2026-04-10

## Context

Reusable agents need a way to declare the external MCP servers they depend on. The question is not only whether agent-local MCP configuration should exist, but also where that configuration should live and what shape it should use.

Key constraints:

- the user wants the configuration colocated with the agent in its directory rather than embedded directly in `AGENT.md`
- Compozy should follow the ecosystem's standard MCP configuration shape instead of inventing a custom schema
- Compozy still needs to inject its own reserved MCP server for nested agent execution, which must not be overridden by agent-local configuration

Current ecosystem practice documented by Claude Code uses a JSON file with a top-level `mcpServers` object for project-scoped MCP configuration. That shape is a better interoperability target than a custom Compozy-specific list schema.

## Decision

**Each agent directory may contain an optional `mcp.json` file that uses the standard MCP configuration shape with a top-level `mcpServers` object.**

Example:

```json
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_TOKEN": "${GITHUB_TOKEN}"
      }
    }
  }
}
```

Compozy-specific rules:

- the filename is `mcp.json` inside the agent directory, not project-root `.mcp.json`
- Compozy parses the standard config shape and converts it into ACP `mcpServers` entries at session setup
- environment placeholders such as `${TOKEN}` are expanded during agent load and session setup; missing variables are validation errors
- the reserved Compozy MCP server name is `compozy` in v1, and it is always injected by the host and cannot be overridden by `mcp.json`
- child agent runs receive only the reserved `compozy` MCP server plus the child agent's own `mcp.json` declarations; parent agent-local MCP servers are not inherited implicitly

## Alternatives Considered

### Alternative 1: Inline MCP declarations in `AGENT.md`

- **Description**: Add MCP configuration directly to the agent frontmatter or metadata block.
- **Pros**: Single-file agent definition; fewer directory artifacts.
- **Cons**: Worse interoperability with existing MCP tooling; harder to reuse or validate against the common MCP config shape; larger mixed-format files.
- **Why rejected**: The user preferred a colocated config file and the standard JSON shape is already familiar in the MCP ecosystem.

### Alternative 2: Use `.mcp.json` inside the agent folder

- **Description**: Keep both the schema and the leading-dot filename used at project root in Claude Code.
- **Pros**: Closer filename parity with project-level conventions.
- **Cons**: The user explicitly asked for `mcp.json` inside the agent folder; using the leading dot adds little value in this context.
- **Why rejected**: Compozy should keep the agent bundle explicit and user-directed while still reusing the standard schema.

### Alternative 3: Global MCP registry referenced by agent name

- **Description**: Let agents reference predeclared MCP profiles defined elsewhere in Compozy config.
- **Pros**: Less duplication across agents.
- **Cons**: Introduces another registry and reference resolution system before the first agent feature is shipped.
- **Why rejected**: This expands scope unnecessarily and weakens portability of an agent directory.

## Consequences

### Positive

- Follows an ecosystem-recognizable MCP schema.
- Keeps the agent bundle self-contained.
- Lets Compozy reuse existing MCP validation and conversion logic more directly.
- Makes it clear which external tools belong to which agent.

### Negative

- Adds a second file to some agent directories.
- Same MCP server may be duplicated across multiple agents until a shared registry exists.

### Risks

- **Risk**: Users may assume the agent file should be named `.mcp.json` because of project-level tools.
  **Mitigation**: Document clearly that Compozy uses `mcp.json` as an agent-local artifact while preserving the standard internal schema.
- **Risk**: Agent-local server names may collide with reserved Compozy MCP server names.
  **Mitigation**: Reserve host-defined names and fail validation when collisions occur.
- **Risk**: Relative executable paths in `mcp.json` could be ambiguous.
  **Mitigation**: Resolve relative paths against the agent directory and document that rule.

## Implementation Notes

- Validate the JSON shape before runtime and surface errors in `compozy agents inspect`.
- Expand environment-variable placeholders such as `${TOKEN}` during agent load and fail closed when a required variable is absent.
- Convert the `mcpServers` object into a deterministic, sorted ACP list before attaching it to `session/new` and `session/load`.
