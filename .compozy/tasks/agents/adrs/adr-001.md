# ADR-001: Flat Filesystem Agent Definitions with Workspace Override

## Status

Accepted

## Date

2026-04-10

## Context

Compozy needs reusable agent definitions that can be shared across workflows, executed from the CLI, and invoked indirectly from ACP-backed runtimes. The reference implementation in agh uses `AGENT.md` files plus additional local capabilities such as per-agent skills, memory, and composition. For Compozy v1, the user explicitly narrowed the scope:

- agent definitions must be discoverable from both workspace and global scopes
- the model should stay flat, without inheritance or composition
- per-agent `skills/` and `memory/` are out of scope for now
- the same agent name must resolve predictably when it exists in multiple scopes

Compozy already has workspace configuration in `.compozy/config.toml`, but it does not have a reusable artifact format for prompt-oriented agent definitions.

## Decision

**Reusable agents will be defined as flat filesystem artifacts rooted at two scopes:**

- workspace: `.compozy/agents/<name>/AGENT.md`
- global: `~/.compozy/agents/<name>/AGENT.md`

Each agent directory may also include an optional `mcp.json` file for agent-local MCP server declarations.

The directory name `<name>` is the canonical agent identifier and must match `^[a-z][a-z0-9-]{0,63}$`. `compozy` is reserved in v1. The file content provides:

- human-facing metadata such as title and description
- runtime defaults such as `ide`, `model`, `reasoning_effort`, and `access_mode`
- the base agent prompt body

`AGENT.md` uses YAML frontmatter followed by a markdown body, matching the repository's existing frontmatter conventions.

Resolution rules:

- workspace scope overrides global scope on same-name conflicts
- no inheritance, `extends`, `uses`, or nested agent composition in v1
- `AGENT.md` and optional `mcp.json` are the only agent-local artifacts loaded in v1
- sibling `skills/` or `memory/` directories are ignored
- when the same agent name exists in both scopes, the winning scope contributes the whole agent directory; Compozy does not merge `AGENT.md` from one scope with `mcp.json` from another

## Alternatives Considered

### Alternative 1: Store agents in `.compozy/config.toml`

- **Description**: Represent reusable agents as TOML sections under workspace configuration.
- **Pros**: Reuses existing configuration loading path; no new artifact type.
- **Cons**: Poor fit for long prompt bodies; makes global distribution awkward; mixes reusable content with command defaults.
- **Why rejected**: Agent prompts are content artifacts, not small scalar settings. Markdown-based artifacts are easier to author, review, and share.

### Alternative 2: Full agh-style agent package in v1

- **Description**: Port `AGENT.md` plus `skills/`, `memory/`, and composition semantics from agh immediately.
- **Pros**: Richer feature set from day one; closer parity with the reference project.
- **Cons**: Larger surface area, more resolution rules, and more prompt assembly complexity before Compozy has proven basic agent reuse.
- **Why rejected**: The user explicitly constrained v1 to a flat model without per-agent skills or memory.

### Alternative 3: Global-only registry

- **Description**: Support only `~/.compozy/agents` and require all reusable agents to be machine-local.
- **Pros**: Simplest discovery logic.
- **Cons**: Makes project-specific agents hard to version with the repository and breaks local reproducibility for teams.
- **Why rejected**: Workspace-scoped agents are required for project-local automation such as council agents tied to repository conventions.

## Consequences

### Positive

- Keeps the first release small and predictable.
- Supports both personal and repository-local agent libraries.
- Reuses a content format that fits prompt-heavy definitions.
- Makes conflict resolution deterministic through workspace-first precedence.

### Negative

- No reuse between agents beyond copy-paste in v1.
- No per-agent local assets such as memory or bundled skills.
- Agent-local MCP configuration becomes part of the agent bundle without requiring a separate global registry.
- Some future migrations may be needed if the schema expands.

### Risks

- **Risk**: Users may expect agh-style `skills/` or `memory/` folders to work.
  **Mitigation**: Validate and warn on unsupported sibling directories instead of silently loading them.
- **Risk**: Two scopes with the same name may surprise users.
  **Mitigation**: Surface source scope in `compozy agents list` and `compozy agents inspect`.
- **Risk**: Users may expect cross-scope MCP merging for same-name agents.
  **Mitigation**: Treat the agent directory as the unit of override and document that behavior explicitly.

## Implementation Notes

- Add a new discovery layer under `internal/core/agents` or equivalent, separate from the ACP transport registry in `internal/core/agent`.
- Parse `AGENT.md` plus optional `mcp.json`, using YAML frontmatter for agent metadata, and treat unknown markdown metadata fields as forward-compatible.
- Include agent source metadata (`workspace` or `global`) in resolved definitions and inspection output.
