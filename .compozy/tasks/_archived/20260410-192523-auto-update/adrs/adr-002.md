# ADR-002: Background Version Check with 24h Cache in main.go

## Status

Accepted

## Date

2026-04-06

## Context

Users should be notified when a new version of Compozy is available without slowing down their workflow. The notification must be non-blocking, respect CI environments, and avoid excessive GitHub API calls. Three integration points were evaluated: `cmd/compozy/main.go`, Cobra `PersistentPreRun`/`PersistentPostRun` hooks, and the `command/command.go` wrapper.

## Decision

Implement the background version check as a goroutine in `cmd/compozy/main.go`, wrapping the `cmd.Execute()` call. Use a YAML state file at `~/.config/compozy/state.yml` (respecting `$XDG_CONFIG_HOME`) with a 24-hour check interval cache.

The pattern:
1. Before `cmd.Execute()`: launch a goroutine that checks for updates (reads state file, optionally queries GitHub API, writes state file)
2. After `cmd.Execute()`: cancel the check context, read the channel, print notification to stderr if a newer version exists
3. Skip the check entirely when `COMPOZY_NO_UPDATE_NOTIFIER` is set or when `version.Version == "dev"`

## Alternatives Considered

### Alternative 1: PersistentPreRun/PersistentPostRun on root command

- **Description**: Add Cobra hooks in `internal/cli/root.go` to launch and collect the version check.
- **Pros**: More Cobra-idiomatic, keeps main.go minimal
- **Cons**: Cobra's `PersistentPreRun` does not fire when a command lacks a `Run` function (known issue #2039). Adds complexity to root.go. All CLI commands become implicitly aware of the update system.
- **Why rejected**: Fragile hook execution behavior and unnecessary coupling to the CLI command layer.

### Alternative 2: Wrapper in command/command.go

- **Description**: Wrap `Execute()` in the public `command` package with update check logic.
- **Pros**: Keeps main.go clean, reusable for embedders
- **Cons**: The `command` package is a thin public wrapper. Adding update logic here expands its responsibility beyond delegation. Embedders may not want update notifications.
- **Why rejected**: Violates the package's single responsibility as a delegation layer.

## Consequences

### Positive

- Zero impact on CLI command code — update check is invisible to all subcommands
- Goroutine runs in parallel with command execution — no latency added
- 24h cache prevents rate limiting and unnecessary network calls
- `COMPOZY_NO_UPDATE_NOTIFIER` provides clean CI opt-out

### Negative

- main.go becomes slightly more complex (10-15 lines added)
- State file requires filesystem access (handled gracefully on failure)

### Risks

- GitHub API rate limiting for unauthenticated requests (60/hour): mitigated by 24h cache
- State file permission issues on restricted systems: mitigated by silently skipping on error

## Implementation Notes

State file path resolution: `$XDG_CONFIG_HOME/compozy/state.yml` (default `~/.config/compozy/state.yml`). This aligns with `internal/setup/agents.go` which already resolves `XDGConfigHome`. The state file stores `checked_for_update_at` (timestamp) and `latest_release` (version, URL, published_at).
