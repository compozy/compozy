# ADR-002: Custom Event Bus with Bounded Per-Subscriber Backpressure

## Status

Accepted

## Date

2026-04-05

## Context

Today `internal/core/run/execution.go` emits job lifecycle messages to a single `chan uiMsg` that only Bubble Tea reads. `internal/core/run/logging.go:HandleUpdate()` uses the same channel to forward ACP session updates. Both publish sites use `select` with a `default` branch, so any slow consumer silently drops events — there are currently seven such drop sites in the executor.

This refactor introduces N subscribers for the same event stream: the existing TUI, a new NDJSON journal writer, future RPC broadcast (Phase B), and future hook dispatcher (Phase C). A single `uiCh` cannot serve this correctly. We need a pub/sub primitive that:

- Fans out typed events to multiple subscribers
- Never blocks the publisher (the ACP session loop and job workers cannot pause for a slow TUI or a laggy RPC client)
- Applies backpressure per-subscriber (slow subscribers degrade alone; they do not starve the journal writer or the TUI)
- Exposes drop metrics so silent loss becomes visible
- Supports clean shutdown (drain all channels, close in order)

The Go ecosystem offers several pub/sub libraries — `asaskevich/EventBus`, `ThreeDotsLabs/watermill`, `cskr/pubsub/v2`, and others. Our library research evaluated them against Compozy's scale (5 subscribers, 100-10k events per run) and our backpressure policy requirements. All were rejected for reasons captured in the Alternatives section.

## Decision

Build a custom generic event bus in `pkg/compozy/events/` (public package), approximately 150 lines of Go. The bus:

- Is generic over event type: `Bus[T any]`
- Allocates a bounded channel per subscriber (default capacity 256)
- Publishes via non-blocking send using a subscriber snapshot (RLock acquired, subscribers copied to a local slice, RLock released, then sends happen without holding the lock)
- Tracks dropped-event counts **per-subscriber** via `atomic.Uint64` embedded in each subscription struct
- Exposes a `Subscribe() (<-chan T, func())` contract where the cleanup func unsubscribes and closes the channel
- Exposes `Close(ctx context.Context) error` that unsubscribes all, closes all subscriber channels, and returns
- Exposes `DroppedFor(subscriptionID) uint64` accessor for observability

The bus sits between the Service Kernel (and its internal components) and downstream consumers:

```
 core.Run() / kernel commands
        │
        ▼
 Journal (assigns seq, writes events.jsonl)   ← ADR-004 says this is upstream
        │
        ▼
 Bus[RunEvent] ──► TUI subscriber
              ──► NDJSON replay subscribers
              ──► Phase B: RPC broadcast subscriber
              ──► Phase C: hook dispatcher subscriber
```

Subscriber registration is thread-safe via `sync.RWMutex`. Publish takes an `RLock`, copies the subscriber handles into a local slice, **releases the lock**, then does non-blocking sends over the snapshot. This keeps publish latency independent of subscriber count and allows subscribers to unsubscribe concurrently with publish without deadlock.

Each subscription is represented by a struct carrying its channel and a per-subscriber `dropped atomic.Uint64`. Drop counters are not aggregated globally; each subscriber's counter is independent and queryable. A rate-limited `slog.Warn` fires when a subscriber drops events (at most 1/sec per subscriber via last-warn timestamp), preventing log floods during sustained degradation.

## Alternatives Considered

### Alternative 1: `github.com/ThreeDotsLabs/watermill`

- **Description**: Full message-routing framework with Kafka, NATS, Redis, and channel backends; middleware, routing, message handlers.
- **Pros**: Battle-tested, actively maintained, wide backend support.
- **Cons**: 30+ transitive dependencies. Framework-scale abstractions (routers, middleware, correlation IDs, message metadata envelope). Designed for distributed message queues, not a single-binary CLI with 5 subscribers.
- **Why rejected**: Wrong scale. Pays cost of distributed-systems architecture for a local-first tool.

### Alternative 2: `github.com/cskr/pubsub/v2`

- **Description**: Small (~200 LOC) generic pubsub with channel-based fan-out.
- **Pros**: Tiny, typed, no deps, MIT.
- **Cons**: Blocks the publisher when any subscriber's channel fills. Publisher blocking is the wrong policy for Compozy — the ACP session and job workers must never pause on TUI or RPC backpressure.
- **Why rejected**: Wrong backpressure policy. Custom cost vs. cskr/pubsub is ~20 lines of difference; that delta is worth owning the policy.

### Alternative 3: `github.com/asaskevich/EventBus`

- **Description**: Generic-free event bus using reflection and string topics.
- **Pros**: Popular (~1.9k stars), simple API.
- **Cons**: No type safety (handlers receive `interface{}`, caller asserts). Uses reflection. Fights Go 1.23 generics.
- **Why rejected**: Type erasure is a regression from where Go is today.

### Alternative 4: NATS embedded (`nats-server/v2`)

- **Description**: Embed NATS server in-process for subject-based pub/sub, request/reply, and streams via JetStream.
- **Pros**: Battle-tested, rich feature set (subject wildcards, JetStream persistent streams).
- **Cons**: ~8-15MB binary bloat; dozens of transitive dependencies (Prometheus client, JWT libs, raft). Engineered for multi-node clustering at millions of messages/sec. Compozy has 5 subscribers, 100-10k events per run, and single-binary philosophy in CLAUDE.md.
- **Why rejected**: Overkill by orders of magnitude; violates single-binary local-first philosophy.

## Consequences

### Positive

- Publisher latency is predictable and independent of subscriber count or subscriber speed.
- Slow subscribers degrade in isolation; the TUI can lag without blocking journal writes.
- Drop metrics are explicit and actionable (counters per-subscriber, exported to logs and future Prometheus scrape).
- ~150 LOC is readable in one sitting — new contributors understand the backpressure policy immediately.
- Generics give compile-time type safety. Each bus is typed to a specific event.

### Negative

- Drops are silent-by-default unless metrics are observed. Mitigation: structured `slog.Warn` on every drop burst, not just metric counters.
- Backpressure policy is hardcoded in the bus (drop on full). Changing it means a code change, not a config change. This is deliberate — backpressure policy is load-bearing and should not be user-configurable in v1.
- No cross-process fan-out. If Phase B's RPC server lives in the same process as the kernel, this is fine; if it ever splits, another mechanism is needed.

### Risks

- **Subscriber leaks**: if a caller forgets to invoke the unsubscribe closure, channels leak. Mitigation: every call site is test-covered; `defer unsub()` is the established pattern.
- **Lock contention at very high subscriber counts**: RWMutex snapshot is O(N) at publish time. With ~5 subscribers this is trivial; if Phase B's RPC server ever fans out to 100+ concurrent clients, shard the bus by `run_id`.
- **Ordering guarantees**: events fan out concurrently to subscribers, so subscribers may observe events slightly out-of-order relative to each other. Inside a single subscriber's channel, order is preserved.

## Implementation Notes

- Package path: `pkg/compozy/events/` (public — consumed by `pkg/compozy/runs/` reader library in ADR-005)
- Public API:
  - `New[T any](bufSize int) *Bus[T]`
  - `(b *Bus[T]) Subscribe() (id SubID, ch <-chan T, unsub func())`
  - `(b *Bus[T]) Publish(ctx context.Context, evt T)`
  - `(b *Bus[T]) Close(ctx context.Context) error`
  - `(b *Bus[T]) DroppedFor(id SubID) uint64`
  - `(b *Bus[T]) SubscriberCount() int`
- Subscription struct holds: `id SubID`, `ch chan T`, `dropped atomic.Uint64`, `lastWarnedAt atomic.Int64` (unix nano, for rate-limited warn)
- Default per-subscriber buffer: 256 events
- Drop logging rate limit: 1 warn/sec per subscriber via CAS on `lastWarnedAt`
- `Close(ctx)` unsubscribes all, closes all subscriber channels, returns after all goroutines drain or ctx expires
- Publisher takes `context.Context` so the publish loop exits during shutdown without leaking
