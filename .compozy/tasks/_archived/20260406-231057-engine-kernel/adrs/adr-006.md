# ADR-006: ACP Ingress Buffer with Grown Capacity and Timed Backpressure

## Status

Accepted

## Date

2026-04-05

## Context

The ACP session implementation at `internal/core/agent/session.go:102` (`sessionImpl.publish`) receives `model.SessionUpdate` messages from the ACP subprocess and forwards them onto a 128-entry buffered channel (`s.updates`). Callers pull from `Updates()` downstream.

The current implementation uses `select` with a `default` clause:

```go
select {
case s.updates <- update:
default:
    // silently drops the update
}
```

Under any consumer slowness — slow journal writes, slow TUI rendering, slow RPC fanout — ACP updates are dropped silently. There is no counter, no log, no signal to operators or tests. This is Point 1 of the two lossy drop sites identified during the code explore. Point 2 (the HandleUpdate/uiCh double-drop in `logging.go`) is resolved by ADR-002's event bus and ADR-004's journal architecture. Point 1 is not; it sits upstream of the kernel entirely.

This matters because the journal (ADR-004) is meant to be the source of truth for a run. If events are lost before reaching the journal, replay is incomplete. The Codex review did not flag this site directly but identified the broader pattern: "the current system is already lossy in live paths. If you keep 'fast-first' and crash before append, you have neither reliable live delivery nor reliable replay."

The ACP subprocess emits session updates at a rate that depends on the AI agent's output characteristics: typically 1-20 events/sec during text generation, with bursts during tool-call phases. 128 entries of headroom is ~6-8 seconds at normal rates, milliseconds during bursts.

## Decision

Grow the `sessionImpl.updates` channel capacity to 1024 and replace the silent `default` drop with timed backpressure followed by explicit drop logging.

**Buffer capacity:** 1024 (8x current). At 20 events/sec sustained burst, that is ~50 seconds of headroom before backpressure engages — enough for any reasonable journal flush cycle or TUI render stall.

**Backpressure policy:** when the buffer is full, the publisher blocks with a 5-second deadline via `select` with ctx-aware cases. If the deadline expires, the event is dropped AND a drop counter is incremented AND a structured warning is logged. The publisher is context-aware so shutdown drains cleanly.

```go
func (s *sessionImpl) publish(ctx context.Context, update model.SessionUpdate) {
    // fast path — non-blocking send
    select {
    case s.updates <- update:
        return
    default:
    }

    // slow path — timed backpressure
    timer := time.NewTimer(5 * time.Second)
    defer timer.Stop()
    select {
    case s.updates <- update:
        s.slowPublishes.Add(1)
    case <-timer.C:
        s.droppedUpdates.Add(1)
        slog.Warn("acp session update dropped after backpressure timeout",
            "session_id", s.id,
            "buffer_cap", cap(s.updates),
            "dropped_total", s.droppedUpdates.Load(),
            "kind", update.Kind,
        )
    case <-ctx.Done():
        return
    }
}
```

**Why 1024 and not unbounded:** unbounded queues can silently accumulate memory during pathological consumer stalls. A bounded queue with explicit backpressure makes the problem visible. 1024 is a balance between "enough headroom for normal bursts" and "obvious signal if something is structurally wrong."

**Why 5 seconds and not 30:** 5 seconds is longer than any expected journal flush (100ms), longer than any TUI render frame (~16ms), and shorter than a human operator's patience for frozen output. It catches transient stalls but surfaces genuine problems quickly.

**Backpressure flow through the ACP subprocess:** when `publish` blocks, the goroutine that reads from the ACP subprocess's stdio also stalls. OS pipe buffers fill (~64KB typically). Eventually the ACP subprocess's writes block, providing natural upstream backpressure. This is the correct behavior — the ACP agent should slow down if we cannot keep up.

## Alternatives Considered

### Alternative 1: Keep 128 buffer, keep silent drop

- **Description**: No change. Accept that events can be lost under load.
- **Pros**: Zero code change.
- **Cons**: Events are silently lost. Journal (ADR-004) claims to be source of truth, but its upstream ingress can drop data with no audit trail.
- **Why rejected**: Directly contradicts ADR-004's goal.

### Alternative 2: Grow to 4096+ with silent drop

- **Description**: Huge buffer, but still silently drop when full.
- **Pros**: Simple; rare to hit the limit.
- **Cons**: Drops become rarer but remain silent. When they do happen, they're invisible. Memory cost is trivial (~1MB) but not the problem.
- **Why rejected**: Silent drops are an observability failure regardless of frequency.

### Alternative 3: Unbounded queue backed by a slice

- **Description**: `[]model.SessionUpdate` with a mutex, unbounded growth.
- **Pros**: Never drops.
- **Cons**: Pathological consumer stalls grow memory without bound. No signal that anything is wrong. In a 20-minute run with a stuck consumer, memory can reach GBs.
- **Why rejected**: Unbounded resources are an anti-pattern for long-running processes.

### Alternative 4: Remove buffer entirely, make publish synchronous

- **Description**: `publish` directly calls the journal synchronously.
- **Pros**: Zero drops. Events are durable by the time publish returns.
- **Cons**: Serializes the ACP reader loop to disk I/O. A stuck journal write freezes the ACP session. Journal flush latency (currently designed at 100ms batches) becomes the ACP latency floor.
- **Why rejected**: Coupling the ACP session loop to journal I/O defeats the batching benefits in ADR-004.

## Consequences

### Positive

- Normal operation: 1024 buffer absorbs bursts, publish is non-blocking on fast path.
- Transient slowness: backpressure makes the publisher wait gracefully, no drops.
- Pathological slowness: drops are explicit, logged, counted, and visible to operators.
- Natural backpressure propagates to the ACP subprocess via OS pipe — the right place to slow down.
- Drop metrics feed into future observability (Prometheus scrape endpoint in Phase B).

### Negative

- Memory cost: 1024 `model.SessionUpdate` pointers ~= ~50KB per session. Trivial.
- Publisher may block for up to 5 seconds on pathological slowness; this is observable in the ACP session loop latency.
- A 5-second stall during `publish` could push the ACP subprocess into a state where it times out its own write; ACP protocol tolerates this but downstream symptoms (session disconnect) may appear.

### Risks

- **Backpressure deadlocks**: if the journal stalls indefinitely AND multiple publishers block on their 5-second timers, the ACP session freezes briefly. Mitigation: context-aware publish; shutdown cancels everyone within timeout.
- **Drop log volume**: during a genuine outage, one log line per dropped event could flood logs. Mitigation: rate-limit drop logs to ~1/sec per session via `slog` attribute-based filtering or a simple `time.Tick`.
- **False sense of durability**: operators may assume no drops = no problems. Mitigation: document that drops indicate structural issues, and expose the counter prominently.

## Implementation Notes

- Change `internal/core/agent/session.go:59` channel capacity from 128 to 1024
- Add `slowPublishes` and `droppedUpdates` `atomic.Uint64` fields to `sessionImpl`
- Change `publish()` signature to accept `context.Context`; update all callers
- Expose metrics via accessor methods on the session: `Session.SlowPublishes() uint64`, `Session.DroppedUpdates() uint64`
- Update session lifecycle logging to emit dropped counts on `Close()`
- Update tests at `internal/core/agent/session_test.go` to cover (a) fast path, (b) backpressure path with slow consumer, (c) timeout path
- This change is LOCAL to the agent package; the kernel, event bus, and journal do not need to know about it — they receive events via `Session.Updates()` as before
