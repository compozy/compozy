# ADR-004: Migration Strategy — v1 → v2 Task Metadata

## Status

Accepted

## Date

2026-04-04

## Context

The existing `compozy migrate` command (`internal/core/migrate.go`) handles the **legacy → v1** migration (XML tags → YAML frontmatter). With ADR-001 (schema v2: drop `scope`/`domain`, add `title`) and ADR-002 (typed enum for `type`), we now need a **v1 → v2** migration path. Existing projects like `.compozy/tasks/acp-integration/` contain task files with the v1 shape:

```yaml
---
status: completed
domain: Agent Runtime
type: Feature Implementation
scope: Full
complexity: high
dependencies: []
---
# Task 1: ACP Agent Layer & Content Model
```

All of these need to become:

```yaml
---
status: completed
title: ACP Agent Layer & Content Model
type: backend        # possibly empty if unmappable
complexity: high
dependencies: []
---
# Task 1: ACP Agent Layer & Content Model
```

We must decide the migration's guessing behavior for the `type` field.

## Decision

Extend `compozy migrate` to run **both** legacy→v1 and v1→v2 passes. The v1→v2 pass does **structural conversion only with best-effort type mapping**:

**Structural changes** (deterministic):
1. Extract `title` from body H1 (`# Task N: <title>` → strip `Task N:` prefix) and insert into frontmatter.
2. Remove `domain` and `scope` fields from frontmatter.
3. Preserve `status`, `complexity`, `dependencies` as-is.

**Type remapping** (best-effort):
- Build a **small case-insensitive mapping table** from common v1 values to v2 types:
  - `"Feature Implementation"` / `"Feature"` → keep empty (require user decision: frontend/backend/etc.)
  - `"Bug Fix"` / `"Bugfix"` / `"Fix"` → `bugfix`
  - `"Refactor"` / `"Refactoring"` → `refactor`
  - `"Configuration"` / `"Config"` / `"Infrastructure"` → `infra`
  - `"Documentation"` / `"Docs"` → `docs`
  - `"Test"` / `"Testing"` → `test`
  - `"Chore"` / `"Cleanup"` → `chore`
  - Exact match (case-insensitive) against the resolved `TypeRegistry` → map to canonical form.
- If no match → set `type: ""` (empty). The file is still **structurally valid v2** but fails `validate-tasks` until the user sets a type.

**Reporting**: The migrate command prints the list of files that need manual `type` correction and emits the fix prompt from ADR-003 for LLM-assisted cleanup.

**Idempotency**: Running migrate on already-v2 files is a no-op (detected by absence of `scope`/`domain` + presence of `title`).

**Detection precedence** (per file):
1. If legacy XML markers (`<task_context>`) present → run legacy→v1 first, then v1→v2 in same pass.
2. Else if `scope` or `domain` present in frontmatter → v1→v2.
3. Else if `title` absent from frontmatter → v1→v2 (title extraction only).
4. Else → skip (already v2).

## Alternatives Considered

### Alternative 1: Pure structural migration, no type mapping at all

- **Description**: Drop scope/domain, extract title, preserve raw `type` string (even if invalid).
- **Pros**: Zero guessing; deterministic.
- **Cons**: Every file fails `validate-tasks` afterward, forcing tedious manual cleanup.
- **Why rejected**: Best-effort mapping reduces manual work significantly without materially increasing risk.

### Alternative 2: Aggressive heuristic (e.g., infer type from `domain` field)

- **Description**: `domain: Frontend` → `type: frontend`; `domain: Database` → `type: backend`, etc.
- **Pros**: Higher automation.
- **Cons**: Domain is free-text and noisy; heuristics will misclassify often.
- **Why rejected**: False positives silently introduce wrong types; user cleanup is harder than starting from empty.

### Alternative 3: Separate command `compozy migrate-v2`

- **Description**: Leave `migrate` alone; add a distinct command for v2.
- **Pros**: Clear separation.
- **Cons**: Two migration commands for users to remember; same logic scattered.
- **Why rejected**: `migrate` is the known entry point. Chained passes keep the UX single-command.

### Alternative 4: Strip `scope`/`domain` but also strip the H1 (frontmatter-only title)

- **Description**: Remove the body H1 during migration.
- **Pros**: Eliminates the sync requirement.
- **Cons**: Conflicts with ADR-001's decision to keep the H1 for human readability.
- **Why rejected**: ADR-001 requires both title locations.

## Consequences

### Positive

- Single `compozy migrate` entry point handles all historical shapes.
- Mechanical conversion reduces user effort; only hard cases require manual attention.
- Idempotent and dry-run-friendly (inherits `--dry-run` from existing command).

### Negative

- Files where `type` can't be mapped end up with empty `type` — technically valid shape but fails runtime validation until fixed.
- The mapping table is an opinion embedded in code; will need occasional updates as the default type list evolves.

### Risks

- **Risk**: Body H1 parsing produces wrong title (author used non-standard header).
  **Mitigation**: Fallback strategies: try `# Task N: X`, then `# Task N - X`, then `# X`. If all fail, write `title: ""` and flag.
- **Risk**: Running migrate on mixed v1/v2 directories produces confusing output.
  **Mitigation**: Per-file detection + clear report line per file (`migrated v1→v2`, `already v2`, `title extraction failed`).

## Implementation Notes

- Add `migrateV1ToV2(path string, content string, registry *TypeRegistry) (*pendingFileMigration, migrationOutcome, error)` to `internal/core/migrate.go`.
- Add title extractor: `extractH1Title(body string) string` with prefix stripping.
- Add type remapping table as a package-level `map[string]string` in `internal/core/tasks/`.
- Extend `MigrationResult` with `UnmappedTypeFiles []string` and `V1ToV2Migrated int` counters.
- When both passes apply, run legacy→v1 first, then v1→v2 in the same write (don't emit intermediate v1 files).
- Fix-prompt output after migration mirrors ADR-003's format for consistency.
