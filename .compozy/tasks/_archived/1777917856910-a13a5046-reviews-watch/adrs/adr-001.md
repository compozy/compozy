# ADR-001: Use a Daemon-Owned Parent Run for Review Watching

## Status

Accepted

## Date

2026-04-30

## Context

The current review remediation loop is manual: an operator waits for CodeRabbit, runs `compozy reviews fetch`, runs `compozy reviews fix`, pushes commits, and repeats until CodeRabbit stops producing feedback. Compozy already has a home-scoped daemon, daemon-backed review fetch/fix flows, durable run state, run streams, cancellation, workspace sync, and provider resolution hooks. The automation needs to survive terminal detaches, expose observable progress, and stay aligned with the daemon-native architecture.

## Decision

Implement `compozy reviews watch` as a daemon-owned parent run with mode `review_watch`. The parent run orchestrates provider polling, review round import, child `reviews fix` runs, optional push, and clean/max-round terminal decisions. Child remediation work continues to use the existing daemon-backed `review` run path instead of introducing a second execution pipeline.

## Alternatives Considered

### Alternative 1: Foreground CLI Loop

- **Description**: Keep all watch state in the `compozy reviews watch` process.
- **Pros**: Lowest initial implementation cost and no new daemon run mode.
- **Cons**: Loses state if the terminal exits, cannot be attached later, duplicates daemon lifecycle logic, and weakens observability.
- **Why rejected**: The workflow is explicitly long-running and should use the daemon's existing ownership model.

### Alternative 2: CI or GitHub Action Automation

- **Description**: Run the review loop in a remote workflow after PR review events.
- **Pros**: Centralized execution and natural proximity to GitHub events.
- **Cons**: Conflicts with Compozy's local-first agent runtime, complicates credentials, and does not reuse local daemon state.
- **Why rejected**: The existing product direction is a local daemon control plane.

## Consequences

### Positive

- The watch loop is durable, cancellable, observable, and attachable.
- Existing fetch/fix/provider behavior remains the remediation source of truth.
- Future daemon UI work can render watch state through existing run APIs.

### Negative

- Adds a new daemon run mode and parent-child orchestration behavior.
- Requires new tests for nested run lifecycle and cancellation.

### Risks

- Parent and child run events can become hard to follow if event names are vague. Mitigate with explicit `review.watch_*` events and child run IDs in payloads.
- Concurrent watches for the same PR can fight over pushes. Mitigate with duplicate active-watch rejection scoped by workspace, provider, and PR.

## Implementation Notes

- Store watch progress in the parent run event journal rather than introducing a new v1 database table.
- Start child review fix runs through the existing daemon `StartReviewRun` path.
- Use the daemon lifecycle context and per-run cancellation; do not create fire-and-forget goroutines.
