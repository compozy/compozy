# ADR-002: Require Provider Watch Status Before Declaring Reviews Clean

## Status

Accepted

## Date

2026-04-30

## Context

`reviews fetch` can return zero items for two different reasons: the provider has reviewed the current PR head and found no actionable issues, or the provider has not produced a review for the new pushed commit yet. The watch loop must not confuse those states, because doing so would stop early and leave a PR unreviewed. CodeRabbit also emits both inline review threads and review-body comments for outside-diff findings, so clean detection must be provider-aware.

## Decision

Extend the provider model with a watch-status capability that reports the PR head SHA, latest provider review identity, latest provider review commit SHA, review state, and enough metadata to distinguish stale, pending, and current reviews. `reviews watch --until-clean` may declare clean only when the provider reports a completed review for the current head and normalized fetch returns zero actionable unresolved items.

## Alternatives Considered

### Alternative 1: Fetch-Only Polling

- **Description**: Repeatedly call `FetchReviews`; stop when it returns zero items.
- **Pros**: Minimal provider API changes.
- **Cons**: Cannot distinguish "not reviewed yet" from "clean"; creates early-exit risk.
- **Why rejected**: It is a timing workaround, not a root-cause fix for provider state ambiguity.

### Alternative 2: GitHub Webhooks

- **Description**: Let GitHub or CodeRabbit events wake the daemon.
- **Pros**: Event-driven and efficient.
- **Cons**: Requires public callback plumbing or a local tunnel, which does not fit local-first v1.
- **Why rejected**: It adds infrastructure outside the current daemon contract.

## Consequences

### Positive

- Clean decisions are based on explicit provider state.
- The watch loop can produce precise operator messages: waiting, stale review, reviewed with issues, or reviewed clean.
- Additional providers can implement the same capability without changing watch orchestration.

### Negative

- Provider implementations need a new optional interface.
- Providers without watch status cannot support `--until-clean`.

### Risks

- CodeRabbit or GitHub response shapes may change. Mitigate with fixture-based tests and clear provider errors.
- Review-body comment parsing can miss new CodeRabbit formats. Mitigate by preserving existing parser coverage and adding fixtures from real review bodies.

## Implementation Notes

- Implement CodeRabbit status using `gh` REST/GraphQL calls, matching existing provider practice.
- Keep `FetchReviews` and `ResolveIssues` unchanged as the issue import and resolution contracts.
- Fail explicitly when a provider lacks watch support instead of guessing.
