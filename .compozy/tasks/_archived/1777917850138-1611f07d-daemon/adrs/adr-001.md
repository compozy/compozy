# ADR-001: Adopt a Global Home-Scoped Singleton Daemon

## Status

Accepted

## Date

2026-04-17

## Context

Compozy currently executes most workflow operations as per-command processes rooted in the active workspace. That makes failover, reattach, multi-client access, and second interfaces harder than they need to be. The approved product direction is to move Compozy toward a single-binary local platform similar in operational posture to AGH while preserving Compozy's workflow model and artifact flow.

The daemon must support:

- a TUI-first terminal experience
- future web access
- richer extension interactions
- robust startup without duplicate daemons
- multiple workspaces managed by one runtime

The user also requires the daemon to always start with `$HOME` as its base, matching AGH.

## Decision

Compozy will adopt one global singleton daemon per user/machine. The daemon will bootstrap from `$HOME`, not from the repository `cwd`, and will manage multiple workspaces through an explicit workspace registry.

The daemon runtime will use a home-scoped operational layout under `~/.compozy`, including singleton runtime files, a global operational database, per-run runtime databases, and logs.

## Alternatives Considered

### Alternative 1: Keep per-command execution and add small helper processes

- **Description**: Preserve the current model and add helper commands or lightweight background workers only where needed.
- **Pros**: Lowest immediate change cost. Reuses current behavior closely.
- **Cons**: Leaves failover, web access, and extension control-plane limitations mostly unsolved. State remains fragmented.
- **Why rejected**: It does not solve the structural problems motivating the redesign.

### Alternative 2: One daemon per workspace

- **Description**: Start a separate daemon for each repository or workspace.
- **Pros**: Keeps runtime state close to the repository. Simpler workspace isolation.
- **Cons**: Creates duplicate daemons, weaker operator ergonomics, and worse multi-workspace coordination. Deviates from the AGH model the team already trusts.
- **Why rejected**: The user explicitly chose a global singleton daemon like AGH.

## Consequences

### Positive

- One control plane can serve CLI, TUI, web, and extensions.
- Startup and reattach become easier to reason about.
- Multi-workspace operations become natural.
- Singleton lock, status, and health can follow AGH patterns closely.

### Negative

- Workspace identity can no longer depend on daemon `cwd`.
- The daemon must maintain a workspace registry and stable path normalization.
- Home-scoped operational state introduces migration work from the old workspace-local run layout.

### Risks

- Incorrect workspace discovery inside the daemon could route operations to the wrong repository.
- Stale lock, socket, or info files can create false-positive health unless bootstrap is idempotent.

## Implementation Notes

- The daemon bootstrap should mirror AGH's `ResolveHomePaths` / `EnsureHomeLayout` setup from `internal/config/home.go`.
- The singleton bootstrap should follow the AGH `internal/daemon/boot.go` flow: lock file, info file, readiness probe, stale PID detection, and stale socket cleanup.
- Auto-start is required, but duplicate daemon creation must be prevented.
- `daemon start` should be idempotent and return success when the daemon is already running.
