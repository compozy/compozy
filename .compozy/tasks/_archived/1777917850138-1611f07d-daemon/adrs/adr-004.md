# ADR-004: Preserve TUI-First UX While Introducing Auto-Start and Explicit Workspace Operations

## Status

Accepted

## Date

2026-04-17

## Context

Users are accustomed to seeing a TUI when they start a workflow run. The daemon redesign must not make the terminal experience feel like a regression. At the same time, the daemon needs explicit workspace operations, attach/watch surfaces, and an auto-start bootstrap model.

The user also chose:

- remove `compozy start`
- keep `compozy sync` and `compozy archive` top-level
- keep TUI as the default interactive experience
- use lazy workspace registration plus explicit workspace commands/endpoints

## Decision

Compozy will preserve a TUI-first interactive run UX while moving execution behind a daemon.

Approved behavior:

- `compozy tasks run <slug>` is the canonical workflow run entrypoint
- interactive TTY sessions attach to the TUI by default
- `--detach`, `--stream`, and `--ui` control attachment mode
- `compozy runs attach <run-id>` reattaches to the interactive client
- `compozy runs watch <run-id>` provides textual live observation
- `compozy sync` and `compozy archive` remain top-level commands

Workspace lifecycle is hybrid:

- workspaces are auto-registered lazily on first touch
- explicit workspace commands and endpoints still exist for operators

## Alternatives Considered

### Alternative 1: Make the daemon visible and mandatory in the user flow

- **Description**: Require operators to start the daemon explicitly and attach to runs only through dedicated follow-up commands.
- **Pros**: Simpler mental split between daemon lifecycle and run lifecycle.
- **Cons**: Feels like a regression from current Compozy UX. Adds ceremony to common flows.
- **Why rejected**: The user wants auto-start and a smooth TUI-first experience.

### Alternative 2: Replace TUI-first interaction with stream-first output

- **Description**: Make textual stream output the default run presentation and treat TUI as optional.
- **Pros**: Simpler detached/runtime model.
- **Cons**: Breaks current expectation and weakens one of Compozy's strongest interactive surfaces.
- **Why rejected**: The user explicitly wants the TUI preserved as the primary experience.

## Consequences

### Positive

- Existing interactive behavior stays recognizable.
- The daemon becomes an architectural change, not a user-facing regression.
- Operators still get explicit workspace management when needed.

### Negative

- CLI, TUI, and daemon lifecycle semantics must be documented carefully.
- Auto-start and attach behavior introduce more bootstrap states to test.

### Risks

- If attach mode defaults are unclear, users may misread detached runs as failures.
- Workspace registry and auto-registration must remain idempotent and path-stable.

## Implementation Notes

- Foreground/background wording is reserved for daemon lifecycle only.
- Same `run_id` concurrent execution is invalid and must be rejected.
- Different runs in the same workspace are allowed to run concurrently.
