# ADR-002: Serve the Embedded SPA from the Daemon's Existing HTTP Listener

## Status

Accepted

## Date

2026-04-20

## Context

The daemon already exposes a localhost-only Gin HTTP server and a UDS server. The user wants the web UI to be the default app that starts with the daemon, and asked for a model close to AGH.

AGH serves its SPA from the same daemon HTTP listener, embedding `web/dist` with `go:embed` and falling back to `index.html` for SPA routes while preserving `/api`.

## Decision

The daemon web UI will be served from the existing daemon HTTP listener:

- SPA assets served at `/`
- daemon APIs preserved at `/api`
- production assets embedded into the Go binary with `go:embed`
- SPA fallback handled by the HTTP server, not by an external frontend process

The UDS server remains API-only for CLI clients.

## Alternatives Considered

### Alternative 1: Run a separate HTTP listener for the web UI

- **Description**: Start a second port or server dedicated to the SPA.
- **Pros**: Separates static and API concerns.
- **Cons**: Adds topology complexity, port coordination, and a split origin model with no clear benefit for a localhost daemon.
- **Why rejected**: The user selected the single-listener model and AGH already proves it works.

### Alternative 2: Keep the SPA external in production

- **Description**: Serve the frontend only through Vite or another external dev server.
- **Pros**: Less Go-side static serving work.
- **Cons**: Breaks the single-binary daemon experience and conflicts with the requirement to ship the web UI as the default app.
- **Why rejected**: The user wants the UI embedded and started with the daemon.

## Consequences

### Positive

- One localhost origin serves both the UI and `/api`.
- Production stays single-binary and local-first.
- Browser deep links work through daemon-owned SPA fallback routing.

### Negative

- `internal/api/httpapi` must grow static serving and NoRoute fallback logic.
- The backend build must ensure the frontend bundle exists before `go build`.

### Risks

- Static fallback routing could shadow API routes if bypass rules are wrong.
- Asset build drift could break production startup if the embed contract is not enforced in the build pipeline.

## Implementation Notes

- Follow AGH's `web/embed.go` and `internal/api/httpapi/static.go` pattern closely.
- Keep `/api` bypass rules explicit.
- Do not add a daemon home directory for unpacked assets unless a later requirement justifies it.
