# ADR-005: Require Full Frontend Verification with Vitest, Playwright, Storybook, and MSW

## Status

Accepted

## Date

2026-04-20

## Context

The user selected the highest validation bar for this feature: strong component and route tests, typed-client contract checks, daemon-served end-to-end coverage for critical flows, and complete Storybook coverage with MSW-backed route and component mocking.

Because this feature introduces a new frontend workspace, daemon static serving, and browser-facing operational controls, weak validation would hide regressions across both Go and web layers.

## Decision

The primary scope includes:

- Vitest coverage for components, hooks, route shells, and domain systems
- tests for generated OpenAPI client integration and error handling
- Playwright flows against the daemon serving embedded assets in production-like mode
- Storybook coverage for routes and reusable components, backed by MSW handlers

Critical E2E coverage must include dashboard, workflows, run detail, reviews, spec, memory, and start/cancel/watch flows.

## Alternatives Considered

### Alternative 1: Vitest-heavy, minimal E2E

- **Description**: Focus on unit and integration tests with only a few browser smoke tests.
- **Pros**: Faster to implement initially.
- **Cons**: Leaves the daemon-served bundle, SPA routing, and real operational flows under-tested.
- **Why rejected**: The user explicitly asked for the complete validation bar.

### Alternative 2: Skip Storybook and mock views only inside tests

- **Description**: Keep mocks in test code only and avoid a dedicated Storybook layer.
- **Pros**: Fewer tools and less setup.
- **Cons**: Weaker review surface for route states, weaker designer/developer feedback, and less parity with AGH's UI workflow.
- **Why rejected**: The user explicitly requested complete Storybook coverage with MSW route and component mocking.

## Consequences

### Positive

- The feature gets coverage across component, integration, daemon-static-serving, and UX review layers.
- Storybook becomes a stable harness for mocking API states without mutating the daemon runtime.

### Negative

- Frontend CI and local verification become heavier.
- The implementation must invest in mock handlers and seeded route fixtures, not just production code.

### Risks

- Incomplete MSW coverage could make Storybook misleading rather than useful.
- E2E flakiness can grow if daemon boot, asset serving, and seeded backend state are not controlled carefully.

## Implementation Notes

- Follow AGH's pattern of Vitest plus Playwright plus Storybook/MSW rather than inventing a lighter custom harness.
- Run Playwright against daemon-served embedded assets, not only Vite dev mode.
- Keep mocked route states colocated with the domain systems they exercise.
