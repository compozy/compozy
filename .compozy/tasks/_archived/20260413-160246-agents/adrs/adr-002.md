# ADR-002: Assemble Agent System Prompts from Resolved Metadata with Explicit Override Precedence

## Status

Accepted

## Date

2026-04-10

## Context

Compozy already separates end-user prompt content from an optional `SystemPrompt` in the ACP execution path. Reusable agents need more than a raw prompt blob:

- each selected agent must inject its own identity and metadata into the system prompt
- the execution context should expose discoverable sibling agents without hardcoding one tool per agent
- `AGENT.md` runtime defaults must influence execution, but explicit CLI flags must still win

Without a dedicated assembly layer, prompt construction would be duplicated across `exec`, workflow runs, and nested agent invocations.

## Decision

**Compozy will introduce an agent prompt assembler that builds the final system prompt from four ordered inputs:**

1. built-in Compozy system framing for the execution mode
2. resolved agent metadata block for the selected agent
3. compact discovery catalog of other available agents
4. the selected agent's prompt body from `AGENT.md`

Runtime configuration precedence for agent-backed executions will be:

1. explicit CLI flags for the current command
2. resolved `AGENT.md` runtime defaults
3. workspace `.compozy/config.toml` command/runtime defaults
4. built-in `model.RuntimeConfig.ApplyDefaults()`

The discovery catalog is intentionally compact in v1. It includes only the fields required for progressive discovery:

- agent name
- short description
- source scope

Detailed metadata stays out of the prompt unless the selected execution path explicitly requests it.

The assembled system prompt has a fixed output shape:

1. built-in Compozy mode framing
2. `<agent_metadata>` block
3. `<available_agents>` block
4. raw markdown body from `AGENT.md`

## Alternatives Considered

### Alternative 1: Treat `AGENT.md` as a literal replacement system prompt

- **Description**: Load the selected agent file and pass its body directly as `SystemPrompt`.
- **Pros**: Very small implementation surface.
- **Cons**: No place to inject source metadata, discovery context, or standardized runtime instructions; higher drift between CLI and nested executions.
- **Why rejected**: Compozy needs consistent prompt framing, not opaque prompt replacement.

### Alternative 2: Let workspace config override agent runtime defaults

- **Description**: Resolve runtime in the same order as today, with agent metadata applied before or below workspace defaults.
- **Pros**: Preserves existing workspace configuration behavior.
- **Cons**: Makes selected agents unreliable because their declared runtime can be silently ignored by workspace defaults.
- **Why rejected**: Once a user explicitly picks an agent, its runtime profile should become the base execution contract unless a CLI flag overrides it.

### Alternative 3: Inject the full agent catalog into every prompt

- **Description**: Dump all available agents and all metadata into the system prompt.
- **Pros**: Maximizes prompt-visible discovery.
- **Cons**: Wastes tokens, scales poorly, and leaks low-value details into every run.
- **Why rejected**: Progressive discovery should stay compact and predictable.

## Consequences

### Positive

- Keeps agent execution deterministic across CLI and nested calls.
- Preserves explicit operator control through CLI flags.
- Creates a reusable assembly seam for future agent-local features.
- Supports progressive discovery without exploding prompt size.

### Negative

- Introduces another prompt-construction layer that must stay aligned with execution modes.
- Requires careful tests for precedence and prompt rendering order.

### Risks

- **Risk**: Prompt duplication between workflow prompts and agent prompts may cause confusing instructions.
  **Mitigation**: Make the assembler additive and mode-aware rather than replacing existing prompt builders wholesale.
- **Risk**: Prompt size may grow noticeably in workspaces with many agents.
  **Mitigation**: Cap catalog content to compact metadata only and sort consistently.

## Implementation Notes

- Reuse the existing `composeSessionPrompt(job.Prompt, job.SystemPrompt)` seam by feeding it an assembled agent system prompt.
- Add snapshot-style prompt rendering tests for `exec --agent`, workflow-driven nested execution, and same-name override scenarios.
- Expose the resolved source scope in the metadata block for debugging and inspection.
- Use `cmd.Flags().Changed(name)` to distinguish explicit CLI overrides from zero values.
