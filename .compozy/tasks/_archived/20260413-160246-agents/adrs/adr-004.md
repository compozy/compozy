# ADR-004: Keep V1 Agent Capability Scope Minimal

## Status

Accepted

## Date

2026-04-10

## Context

The agh reference includes a richer agent model with per-agent skills, memory assembly, and compositional features. Compozy could adopt all of that now, but the immediate user requirement is narrower:

- define reusable agents in global and workspace scopes
- execute them via CLI
- invoke them from ACP-backed runtimes through tools
- inject agent metadata and progressive discovery into system prompts

Expanding beyond that scope would increase implementation time and create more unresolved semantics before the first release proves useful.

## Decision

**Compozy v1 will ship only the minimal capability set needed for reusable agent execution:**

- flat `AGENT.md` definitions
- optional agent-local `mcp.json` using the standard MCP config shape
- runtime defaults attached to the agent
- system prompt assembly with agent metadata and compact discovery
- CLI inspection and execution
- ACP-backed nested execution through a generic tool

The following features are explicitly deferred:

- per-agent `skills/`
- per-agent `memory/`
- inheritance
- composition
- agent-local dependency graphs

## Alternatives Considered

### Alternative 1: Include per-agent skills in v1

- **Description**: Let each agent expose local `skills/` folders resolved during execution.
- **Pros**: Richer agent identity; closer to the reference project.
- **Cons**: Requires setup/preflight changes, skill merging semantics, and new conflict rules.
- **Why rejected**: The user explicitly removed this from v1 scope.

### Alternative 2: Include per-agent memory in v1

- **Description**: Give each agent its own memory files and injection layer.
- **Pros**: Better continuity for long-lived agent personas.
- **Cons**: Raises unresolved persistence, scoping, and compaction semantics; duplicates existing workflow memory concerns.
- **Why rejected**: Not required to make reusable agents operational in Compozy.

### Alternative 3: Add inheritance and composition immediately

- **Description**: Support `extends`, `uses`, or linked agents from day one.
- **Pros**: More reuse and less prompt duplication.
- **Cons**: More complicated precedence, cycle detection, and debugging.
- **Why rejected**: The user explicitly asked for a flat model.

## Consequences

### Positive

- Keeps the initial implementation focused on the core execution problem.
- Reduces parsing, resolution, and testing complexity.
- Leaves room to grow later without blocking the first useful release.

### Negative

- Some prompt duplication across agents is expected.
- Future enhancements may require schema extensions and migrations.

### Risks

- **Risk**: Users may immediately ask for features that were intentionally deferred.
  **Mitigation**: Document the deferred items clearly in the TechSpec and CLI help.
- **Risk**: The minimal schema may be too narrow for some early adopters.
  **Mitigation**: Preserve unknown metadata fields and keep the parser forward-compatible.

## Implementation Notes

- Validation should explicitly reject unsupported fields such as `extends` rather than silently ignoring them.
- `compozy agents inspect` should surface deferred capability markers so future evolution remains visible.
