# ADR-001: Canonical Daemon Transport Contract

## Status

Accepted

## Date

2026-04-20

## Context

The current daemon transport layer keeps JSON-facing DTOs and SSE details spread across `internal/api/core/interfaces.go`, handlers, the daemon client, and `pkg/compozy/runs`. Some handlers return anonymous response shapes, the client applies path-specific decoding logic, and the public run reader package still carries transport knowledge that can drift from handler behavior. The analysis pack identifies this as the main structural boundary problem in the current daemon.

The approved scope for this TechSpec keeps the daemon architecture as the baseline, but allows one deeper structural refactor where the boundary is wrong. The user chose transport and API contracts as that boundary.

## Decision

Introduce `internal/api/contract` as the canonical daemon contract package.

This package becomes the single source of truth for:

- transport DTOs used by daemon handlers and `apiclient`
- response envelopes and error payloads
- SSE event shapes, cursor semantics, heartbeat, and overflow records
- run snapshot and stream payloads consumed by `pkg/compozy/runs`

Handlers, the daemon client, SSE helpers, and `pkg/compozy/runs` will converge on this shape. Adapters are allowed only where public API stability requires them.

## Alternatives Considered

### Alternative 1: Keep DTOs inline in `internal/api/core`

- **Description**: Continue defining transport payloads in `internal/api/core/interfaces.go` and let handlers and clients decode ad hoc response shapes.
- **Pros**: Lowest immediate change count.
- **Cons**: Keeps transport drift and duplicated knowledge in place; makes parity and contract testing weaker.
- **Why rejected**: It preserves the exact architectural problem the analysis highlights.

### Alternative 2: Separate daemon-internal and public contracts from the start

- **Description**: Create one internal transport contract and a second public contract layer for `pkg/compozy/runs` and future UI clients.
- **Pros**: Strong boundary separation.
- **Cons**: Doubles mapping work and increases maintenance cost before the daemon contract is even stable.
- **Why rejected**: Too much ceremony for the current stage; adapters at the public edge are enough.

## Consequences

### Positive

- One canonical transport shape for HTTP, UDS, SSE, `apiclient`, and `pkg/compozy/runs`
- Clear foundation for parity tests and contract-focused fixtures
- Easier future UI work because the daemon payload model is explicit

### Negative

- Migration touches handlers, client decoding, stream helpers, and public run readers together
- Some temporary adapter code may be needed while internal call sites converge

### Risks

- `pkg/compozy/runs` may accidentally become too tightly coupled to daemon-internal implementation details
- SSE contract changes can break stream consumers if cursor and heartbeat semantics are not frozen carefully

## Implementation Notes

- Start by extracting existing DTOs and envelopes from `internal/api/core/interfaces.go`
- Freeze SSE cursor, heartbeat, overflow, and error semantics in the new contract package before refactoring consumers
- Keep handler/service interfaces transport-neutral; only the payload types move into `internal/api/contract`
