# ADR-004: Observability as a First-Class Daemon Contract

## Status

Accepted

## Date

2026-04-20

## Context

The analysis pack shows that the daemon currently has thin observability despite already persisting strong raw runtime data. Health is minimal, metrics expose only a few counters, the daemon log file path exists but is unused, and transcript projections do not yet assemble into a canonical replay surface. Because the approved transport contract refactor will already touch snapshots, SSE, and public run reading, observability can be strengthened in the same boundary rather than treated as an unrelated concern.

The user explicitly chose observability as a first-class part of this same TechSpec.

## Decision

Include richer daemon observability in the primary scope.

The contract must cover:

- a more informative `/daemon/health` response
- a useful `/daemon/metrics` surface with runtime and reconciliation counters
- strong `GET /runs/:run_id/snapshot` payloads for attach and inspection
- canonical transcript assembly from persisted run data

## Alternatives Considered

### Alternative 1: Keep observability narrow and focus only on transport/runtime correctness

- **Description**: Make minimal health and metrics changes while prioritizing the transport refactor and lifecycle fixes.
- **Pros**: Smaller immediate scope.
- **Cons**: Leaves operators with weak visibility into the very systems being refactored.
- **Why rejected**: It undercuts the value of the new contract and makes runtime hardening harder to debug.

### Alternative 2: Add health and metrics now, defer transcript assembly

- **Description**: Improve daemon status endpoints first and treat transcript replay later.
- **Pros**: Reduces the initial replay-model workload.
- **Cons**: Leaves the public run-reader boundary incomplete and preserves a known cold-replay gap.
- **Why rejected**: Snapshot and replay are part of the same user-facing daemon inspection story.

## Consequences

### Positive

- Operators gain a real inspection surface for daemon and run state
- The new transport contract covers both control and observability concerns coherently
- Transcript replay becomes stable enough for headless tooling and CI artifacts

### Negative

- The scope now includes both write-side and read-side improvements
- Observability changes may require extra aggregation logic beyond raw event persistence

### Risks

- Snapshot payloads can become too heavy if not bounded carefully
- Transcript assembly can drift from live UI semantics if the canonical model is underspecified

## Implementation Notes

- Reuse persisted `run.db` projections where possible rather than rebuilding everything from raw logs on every request
- Bound snapshot and transcript payload sizes explicitly
- Surface degraded-health reasons as structured details, not only booleans
