# ADR-004: Mode-Specific System Prompts

## Status

Accepted

## Date

2026-03-31

## Context

Compozy operates in multiple execution modes (PRD task execution, PR review remediation) each with distinct behavioral requirements for the agent. Previously, a reasoning-effort hint and a team directive were appended via `--append-system-prompt`. With the shift to interactive mode, the system prompt becomes the primary control mechanism for agent behavior — it must include mode-specific instructions, skill references, and the job completion signaling protocol.

## Decision

Use `--system-prompt` flag (full system prompt, replacing `--append-system-prompt`) with mode-specific prompt builders. Each mode gets a tailored system prompt via `prompt.BuildSystemPrompt(mode, jobID, signalPort)` that includes:

1. Mode-specific behavioral instructions and skill references
2. Job completion signaling protocol (`curl POST localhost:<port>/job/done`)
3. Reasoning effort directives

The composer message stays short (file path reference only), while the system prompt carries all behavioral context.

## Alternatives Considered

### Alternative 1: Keep --append-system-prompt with longer composer messages

- **Description**: Append minimal system instructions, put mode-specific content in the composer prompt
- **Pros**: Less change to current approach
- **Cons**: Composer messages become long and complex, composer simulation is slower for long text, mixes behavioral instructions with task content
- **Why rejected**: Clean separation (system prompt = behavior, composer = task) is architecturally cleaner and faster

### Alternative 2: CLAUDE.md file per mode

- **Description**: Write mode-specific instructions to a `.claude/CLAUDE.md` file before starting each job
- **Pros**: Native Claude Code mechanism, no flag needed
- **Cons**: File modification between jobs introduces race conditions, shared directory conflicts with concurrent jobs, less explicit than CLI flag
- **Why rejected**: Flag-based injection is more explicit, isolated per process, and doesn't require filesystem coordination

## Consequences

### Positive

- Clean separation: system prompt = agent behavior, composer = task reference
- Each mode gets precisely tailored instructions
- Job-done signaling protocol is baked into every system prompt
- Easy to add new modes (just add a new prompt builder case)
- Skill references per mode improve agent performance

### Negative

- System prompt string can become large — Claude Code must handle it
- Changes from `--append-system-prompt` to `--system-prompt` means replacing rather than extending default behavior
- Mode-specific prompts need maintenance as workflows evolve

### Risks

- System prompt too long for CLI arg: mitigated by OS arg limit (~256KB on macOS), prompts are well under this
- `--system-prompt` flag behavior differs from `--append-system-prompt`: verified that `--system-prompt` sets the full system prompt for the session

## Implementation Notes

- `prompt.BuildSystemPrompt()` is the single entry point; it switches on `model.ExecutionMode`
- The job-done curl command is always appended to every mode's prompt as the last section
- Signal port is injected at build time (not hardcoded) to support `--signal-port` flag
- Reasoning effort hints move from `--append-system-prompt` into the mode-specific prompt content
