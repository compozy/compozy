# ADR-002: Fiber HTTP Server for Job Signaling

## Status

Accepted

## Date

2026-03-31

## Context

With interactive PTY-based execution, the previous job completion detection (exit code from `cmd.Run()`) no longer applies — Claude Code stays alive after completing a task, waiting for further user input. The system needs a deterministic mechanism for the agent to signal "I'm done with this job" so Compozy can advance to the next job in the queue.

The agent is instructed via system prompt to execute a signaling command upon task completion. The question is how Compozy receives that signal.

## Decision

Run a local Fiber HTTP server on `localhost:9877` (configurable via `--signal-port`). The agent signals job completion by executing `curl -s -X POST http://localhost:9877/job/done -H 'Content-Type: application/json' -d '{"id":"<job-id>"}'`. The server validates the job ID, sends a `SignalEvent` on an internal Go channel, and the Bubbletea UI reacts by marking the job as done and switching to the next pending job.

Endpoints: `POST /job/done` (completion signal), `POST /job/status` (optional progress updates), `GET /health` (connectivity check).

## Alternatives Considered

### Alternative 1: File-based sentinel (fsnotify)

- **Description**: Agent creates `.tmp/jobs/<id>.done` file; Compozy watches via fsnotify
- **Pros**: No TCP port needed, simple
- **Cons**: fsnotify has platform quirks (macOS kqueue limits), adds another dependency, race conditions with file creation timing, harder to debug
- **Why rejected**: Less reliable across platforms, harder to extend with structured data

### Alternative 2: Unix socket / named pipe

- **Description**: Compozy creates a Unix socket; agent writes to it
- **Pros**: No TCP port, slightly more secure
- **Cons**: Platform-specific (no Windows support), harder for agent to interact with (`curl` doesn't work with Unix sockets easily), less debuggable
- **Why rejected**: Agent needs to `curl` from within Claude Code — Unix sockets make this awkward; Compozy is local-first anyway

### Alternative 3: MQTT broker

- **Description**: Run a local MQTT broker for pub/sub messaging between agents and Compozy
- **Pros**: Rich pub/sub semantics, message persistence
- **Cons**: Massive overkill for local IPC, adds broker dependency, complex setup
- **Why rejected**: This is single-machine, single-user communication — MQTT solves a distributed problem we don't have

## Consequences

### Positive

- Debuggable: `curl localhost:9877/job/done` works from any terminal
- Proven: agh POC uses the same Fiber-based approach successfully
- Extensible: easy to add endpoints for progress updates, status queries, future features
- No platform-specific code
- Agent interaction is a simple curl command (easy to instruct via system prompt)

### Negative

- Opens a TCP port on localhost (ephemeral, but could conflict)
- Adds Fiber as a dependency
- Agent must have network access to localhost (always true in practice)

### Risks

- Port conflicts: mitigated by `--signal-port` flag and clear error on bind failure
- Agent doesn't send signal: mitigated by existing activity timeout as safety net
- Security: localhost-only, no auth needed; port is ephemeral and dies with process

## Implementation Notes

- Server starts before any jobs launch, shuts down after all jobs complete
- Use `fiber.Config{DisableStartupMessage: true}` to avoid noisy stdout
- Event channel should be buffered (`chan SignalEvent`, cap = number of jobs) to avoid blocking the HTTP handler
- Validate job ID against known jobs; respond 404 for unknown IDs
