# ADR-001: Model Ad Hoc Execution as a First-Class Runtime Mode

## Status

Accepted

## Date

2026-04-05

## Context

Compozy currently supports two execution flows: PR review remediation and PRD task execution. Both flows rely on the same runtime stack: CLI config parsing, workspace defaults, ACP runtime validation, prompt artifact generation, and the shared executor in `internal/core/run`.

The requested `compozy exec` command introduces a third kind of execution: running an ad hoc prompt with explicit runtime settings such as `--ide`, `--model`, and output format controls. A naive implementation could bypass the existing preparation/execution flow and create a dedicated command path that talks directly to ACP runtimes. That would duplicate runtime validation, retry behavior, logging, output handling, and artifact generation.

The system already shows one architectural weakness: preparation logic is tightly coupled to directory-backed workflows. This limitation should be addressed by generalizing the pipeline, not by adding a side path.

## Decision

Implement `compozy exec` as a third first-class execution mode that reuses the existing configuration, preparation, and execution pipeline.

The implementation will:
- Add a dedicated execution mode for ad hoc runs to the shared runtime model.
- Extend preparation to support prompt-backed runs in addition to directory-backed task/review runs.
- Keep `run.Execute`, ACP client bootstrap, retries, logging, and summary handling shared across all modes.
- Avoid a direct command-to-ACP implementation path outside the shared pipeline.

## Alternatives Considered

### Alternative 1: Add a thin standalone `exec` command that creates one job outside the planner

- **Description**: Build a single ad hoc job directly inside the CLI command and hand it to the executor.
- **Pros**: Smaller initial diff, fewer changes to planner input resolution.
- **Cons**: Splits job construction across two code paths, duplicates artifact decisions, and encourages more command-specific drift later.
- **Why rejected**: It solves the immediate feature but preserves the root problem that job preparation is not modeled as a reusable capability.

### Alternative 2: Talk directly to ACP runtimes from `compozy exec`

- **Description**: Build a dedicated command that bypasses the planner and executor, managing runtime I/O itself.
- **Pros**: Maximum local freedom for the command.
- **Cons**: Duplicates retries, logging, session handling, output formatting, and future runtime policy changes.
- **Why rejected**: This would create the exact workaround-style parallel stack the feature should avoid.

## Consequences

### Positive

- Keeps runtime behavior consistent across `start`, `fix-reviews`, and `exec`.
- Makes future execution modes cheaper because preparation and artifact policies become explicit shared concepts.
- Prevents command-specific ACP drift and duplicated failure handling.

### Negative

- Requires a deeper refactor than a command-local implementation.
- Forces planner and runtime model APIs to handle both directory-backed and prompt-backed inputs.

### Risks

- Refactoring shared preparation code may regress existing `start` or `fix-reviews` flows if not covered well.
- Mode branching can become messy if ad hoc execution introduces too many special-case fields.

## Implementation Notes

- Prefer adding a small ad hoc preparation path to existing planning code over creating a new package prematurely.
- Keep the execution contract job-based: ad hoc execution still produces one or more `model.Job` values plus run metadata.
- Add regression tests for all three modes before considering the refactor complete.
