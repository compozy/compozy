# ADR-003: Support Multi-Source Prompt Input and Structured Output for `compozy exec`

## Status

Accepted

## Date

2026-04-05

## Context

The `exec` command is intended for ad hoc execution, but "ad hoc" covers different usage patterns:
- short prompts typed directly in the shell
- long prompts stored in files
- piped prompts from other tooling

The command also needs to support two different output audiences:
- humans running the command interactively
- automation that needs machine-readable results

The repository already uses workspace config precedence of `flags > command section > [defaults] > internal defaults`. The new command should follow that rule rather than invent its own precedence model.

## Decision

Design `compozy exec` with:
- prompt input from a positional argument, `--prompt-file`, or `stdin`
- explicit `--format text|json`
- workspace config precedence `flags > [exec] > [defaults] > internal defaults`

In JSON mode, the command will emit a machine-readable run result and suppress the human/TUI presentation path. In text mode, it will keep the normal human-oriented runtime presentation.

## Alternatives Considered

### Alternative 1: Positional prompt only

- **Description**: Require `compozy exec "<prompt>"` for all executions.
- **Pros**: Small CLI surface.
- **Cons**: Poor fit for long prompts, generated prompts, and Unix pipelines.
- **Why rejected**: It makes automation and reproducibility unnecessarily awkward.

### Alternative 2: Text-only output with no JSON contract

- **Description**: Reuse current human output and leave automation to log scraping.
- **Pros**: Minimal executor changes.
- **Cons**: Forces machine consumers to parse unstable human text and makes `exec` weak for scripting.
- **Why rejected**: The user explicitly requested output options, and log scraping is the wrong abstraction.

## Consequences

### Positive

- Makes the command ergonomic for humans and automation.
- Preserves the repository's existing config precedence model.
- Keeps output format selection explicit and testable.

### Negative

- Input validation becomes more complex because prompt sources must be resolved predictably.
- JSON mode requires a stable result schema and explicit suppression of UI-oriented output.

### Risks

- Ambiguous combinations of positional prompt, `--prompt-file`, and piped stdin could confuse users if precedence is not strict.
- JSON consumers may depend on fields that are not versioned if the schema is underspecified.

## Implementation Notes

- Reject ambiguous prompt-source combinations instead of guessing silently.
- Keep the JSON schema focused on run metadata, artifact paths, status, usage, and per-job summaries.
- Add CLI and integration tests for all prompt-source combinations and JSON mode behavior.
