# ADR-003: Force Auto-Commit and Allow Dirty Worktrees for Auto-Push Watch Runs

## Status

Accepted

## Date

2026-04-30

## Context

The target workflow is mechanical: after each review-fix round, the agent creates commits and the operator pushes them so CodeRabbit reviews the next head. The user explicitly chose an auto-push policy where `auto_commit` is always enabled and the watch loop does not block on a dirty worktree. The repository policy forbids destructive git commands without explicit user permission.

## Decision

When `--auto-push` is enabled, `reviews watch` forces `auto_commit=true` and rejects `--auto-commit=false`. The watch run accepts dirty worktrees. It never runs `git restore`, `git checkout`, `git reset`, `git clean`, `git rm`, or manual staging commands. It only pushes committed branch state after a successful child fix run and emits warnings when existing unpushed commits are included.

## Alternatives Considered

### Alternative 1: Clean Worktree Required

- **Description**: Refuse to start auto-push watch runs unless `git status` is clean.
- **Pros**: Stronger isolation between prior user edits and watch-generated changes.
- **Cons**: Does not match the requested workflow and blocks legitimate active work.
- **Why rejected**: The user chose dirty-worktree tolerance.

### Alternative 2: Manual Push by Default

- **Description**: Fix rounds automatically but require the operator to push.
- **Pros**: Safer for repositories with complex branch policy.
- **Cons**: Leaves the main manual bottleneck in place.
- **Why rejected**: The goal is to remove repeated mechanical waiting and pushing.

## Consequences

### Positive

- The loop can run end-to-end without operator intervention between rounds.
- The implementation respects repository data-loss constraints.
- Push behavior is explicit and observable.

### Negative

- Existing unpushed commits can be pushed along with watch-generated commits.
- The watch loop cannot guarantee that every pushed commit was created by the current round.

### Risks

- A child fix run may complete without creating a commit. Mitigate by requiring HEAD to advance after non-empty rounds before pushing.
- Upstream branch configuration may be missing. Mitigate with explicit `--push-remote` and `--push-branch` overrides and clear validation errors.

## Implementation Notes

- Use a small git runner abstraction that only supports read commands and `git push`.
- Emit structured events for initial dirty state, unpushed commits, push start, push success, and push failure.
- Do not weaken agent prompts that already say automatic commits must follow clean verification.
