# ADR-003: Expose the Daemon Through AGH-Aligned REST Transports Using Gin

## Status

Accepted

## Date

2026-04-17

## Context

The daemon needs operator-facing transports for CLI, TUI, web, and future integrations. The team already has a mature reference in AGH, which exposes daemon services through REST-style endpoints and SSE using `gin` over both UDS and TCP HTTP.

The user explicitly asked not to invent a new command-style HTTP API and wants the same web framework used by AGH.

## Decision

Compozy will expose daemon services through AGH-aligned REST endpoints implemented with `gin`.

The transport posture is:

- UDS for CLI
- HTTP on localhost for local web and client surfaces
- SSE for streaming and observation
- TCP disabled by default
- no localhost auth/token in v1

This decision applies only to operator-facing daemon transports. In-process service boundaries remain typed Go interfaces. Extension subprocess boundaries remain JSON-RPC where that already fits. Agent runtime boundaries remain ACP.

## Alternatives Considered

### Alternative 1: Command-oriented HTTP API

- **Description**: Expose routes that mirror CLI verbs such as `/commands/tasks.run`.
- **Pros**: Direct mapping from CLI to API.
- **Cons**: Weaker resource model, poorer web ergonomics, and unnecessary divergence from AGH.
- **Why rejected**: The user explicitly rejected inventing a new endpoint style.

### Alternative 2: JSON-RPC everywhere

- **Description**: Use JSON-RPC for all daemon communication surfaces.
- **Pros**: Uniform RPC model.
- **Cons**: Bad fit for browseable REST resources, SSE observation, and AGH-aligned transport reuse.
- **Why rejected**: The approved contract split is layered, not JSON-RPC everywhere.

## Consequences

### Positive

- The daemon API matches an internal pattern the team already knows.
- Web/TUI/CLI clients can share a stable resource-oriented model.
- SSE fits naturally for watch and observe flows.
- Reusing `gin` reduces framework churn and design noise.

### Negative

- The TechSpec must define route conventions carefully so CLI and web remain consistent.
- No auth on localhost is a deliberate v1 trade-off and should be documented as local-only.

### Risks

- If localhost exposure ever expands accidentally beyond local binding, the no-auth posture becomes unsafe.
- Diverging from AGH's resource design later would create migration noise for clients.

## Implementation Notes

- Prefer REST resources for workspaces, workflows, runs, reviews, and daemon status.
- Use explicit action subresources only when REST resource mutation is not expressive enough.
- Keep UDS and localhost HTTP contracts aligned through a shared handler core, following AGH's `internal/api/core/interfaces.go`, `internal/api/core/handlers.go`, and `internal/api/core/sse.go`.
- Follow AGH's near-parity route registration pattern from `internal/api/httpapi/routes.go` and `internal/api/udsapi/routes.go`.
