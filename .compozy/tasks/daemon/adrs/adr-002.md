# ADR-002: Keep Human Artifacts in the Workspace and Move Operational State to Home-Scoped SQLite

## Status

Accepted

## Date

2026-04-17

## Context

Compozy workflows are built around human-readable Markdown artifacts such as PRDs, TechSpecs, tasks, reviews, ADRs, and memory files. Those artifacts are valuable because they can be created, edited, and reviewed independently of the daemon or even independently of Compozy.

At the same time, the current run model stores operational state under workspace-local `.compozy/runs/`, which is useful for debugging but weak as a long-lived source of truth for runtime state, attach/replay behavior, and multi-client coordination.

The approved direction is to preserve workflow artifacts while upgrading operational robustness.

## Decision

Compozy will keep human-authored workflow artifacts in the workspace under `.compozy/tasks/<slug>/`, but all operational state will move to `~/.compozy` and be stored in SQLite.

The storage split is:

- `~/.compozy/db/global.db` for global operational and registry state
- `~/.compozy/runs/<run-id>/run.db` for per-run events, transcript, and replayable execution state

All human Markdown artifacts remain syncable during active runs. The daemon will watch only the active run/workflow scope, not the entire machine, and will propagate fresh Markdown state into the database and UI surfaces.

Legacy workspace-local `.compozy/runs/` runtime data will not be imported or maintained for compatibility. The cutover is hard.

`_tasks.md` and `_meta.md` do not remain in the daemonized model.

## Alternatives Considered

### Alternative 1: Move everything into SQLite

- **Description**: Store workflow artifacts and operational state only in the database.
- **Pros**: Single source of truth. Strong queryability.
- **Cons**: Breaks Compozy's flexible Markdown-centric workflow model and makes external artifact generation much harder.
- **Why rejected**: The user explicitly wants Markdown artifacts preserved as first-class workflow documents.

### Alternative 2: Keep operational state in workspace-local `.compozy/runs/`

- **Description**: Add a daemon but continue storing runtime state primarily in workspace files.
- **Pros**: Reuses current readers and file layout closely.
- **Cons**: Weak operational truth, awkward multi-client coordination, and poor long-term daemon ergonomics.
- **Why rejected**: It does not deliver the desired robustness.

## Consequences

### Positive

- Human workflow artifacts remain portable and editable.
- Operational state becomes durable, queryable, and replay-friendly.
- Web and TUI clients can reflect live Markdown changes without re-parsing the whole workspace continuously.
- Cutover remains simple because old runtime logs are not treated as valuable domain state.

### Negative

- Sync semantics between Markdown and DB must be explicit.
- Public run readers that assume workspace-local `.compozy/runs/` need redesign.
- The daemon must manage watchers carefully to avoid waste.

### Risks

- Poorly defined sync semantics could cause surprising UI state.
- Migrating code that assumes `workspaceRoot/.compozy/runs` is a sensitive seam.

## Implementation Notes

- Workflow slug uniqueness remains enforced among non-archived workflows.
- Different `run_id`s may execute concurrently.
- The same `run_id` must never execute in parallel.
- Active-run watchers must stay scoped to the current run/workflow.
- The global DB and per-run DB split should follow AGH's `globaldb` and `sessiondb` architecture, adapted to Compozy's domain.
- The per-run DB writer should borrow AGH's dedicated writer-loop pattern from `internal/store/sessiondb/session_db.go`.
