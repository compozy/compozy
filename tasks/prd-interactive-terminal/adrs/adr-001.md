# ADR-001: PTY + VT Emulator for Terminal Embedding

## Status

Accepted

## Date

2026-03-31

## Context

Looper currently executes Claude Code in headless mode (`--print --output-format stream-json`), capturing stdout/stderr via `io.MultiWriter` and parsing JSON for token usage and log display. This provides structured data but zero observability into what the agent is actually doing — users see formatted log lines, not the real Claude Code TUI with tool calls, edits, and reasoning in real-time. Users also cannot interact with running instances to redirect, clarify, or intervene.

The goal is to embed live, interactive Claude Code TUI sessions inside Looper's existing Bubbletea UI, allowing both real-time observation and direct interaction.

## Decision

Use `charmbracelet/x/xpty` for pseudo-terminal management and `charmbracelet/x/vt` (SafeEmulator) for virtual terminal emulation. Each job gets its own PTY + VT emulator pair. The PTY spawns Claude Code in interactive mode (no `--print`). A goroutine reads PTY output and feeds it to the VT emulator. The Bubbletea UI renders the emulator's screen buffer via `emu.Render()`. User keystrokes are translated and forwarded to the PTY for direct interaction.

## Alternatives Considered

### Alternative 1: Enhanced stream-json rendering

- **Description**: Keep headless mode but parse JSON events more richly — show tool calls, diffs, thinking in a styled Bubbletea view
- **Pros**: No new dependencies, keeps structured data, simpler architecture
- **Cons**: No interaction possible, lossy representation of Claude's actual output, significant custom rendering work for each message type
- **Why rejected**: Fundamentally cannot provide interaction, and the rendering effort approaches that of a terminal emulator anyway

### Alternative 2: tmux-based embedding (like agh POC)

- **Description**: Use tmux to manage Claude Code sessions, with Bubbletea as a control overlay
- **Pros**: Proven in POC, Claude Code renders natively in tmux pane
- **Cons**: Adds tmux as hard dependency, breaks unified UX (must switch between Bubbletea and tmux), no programmatic access to terminal content
- **Why rejected**: Dependency too heavy for a CLI tool, UX fragmentation unacceptable

### Alternative 3: tea.Exec() for subprocess handoff

- **Description**: Use Bubbletea's built-in `tea.Exec()` to hand terminal control to Claude Code
- **Pros**: Zero complexity, native Bubbletea support
- **Cons**: Pauses entire Bubbletea program — no sidebar, no multi-job management, no concurrent execution
- **Why rejected**: Cannot maintain the multi-job dashboard while an agent runs

## Consequences

### Positive

- Full real-time observability of agent work (tool calls, edits, reasoning)
- Direct user interaction with running agents
- Completed job terminals remain navigable (scrollback)
- Stays within Charm ecosystem (xpty, x/vt, bubbletea)
- Proven pattern — 5+ production projects use this exact approach

### Negative

- Loses structured stream-json data (token usage, typed messages)
- Depends on pre-v1 library (x/vt) with no stability guarantees
- More complex architecture (PTY lifecycle, response forwarding, thread safety)
- Potential rendering issues with Claude Code's Ink-based TUI

### Risks

- x/vt breaking changes: mitigated by pinning pseudo-version and wrapping in Terminal abstraction
- Claude Code ANSI sequences not fully supported: mitigated by x/vt's comprehensive VT220+ support; test early with real sessions
- Memory growth from many open terminals: mitigated by VT scrollback limits (configurable)

## Implementation Notes

- Use `vt.NewSafeEmulator()` (not `NewEmulator()`) for thread safety between PTY reader goroutine and Bubbletea render loop
- Response forwarding (`io.Copy(pty, emu)`) is mandatory — without it, Claude Code deadlocks on cursor position queries
- PTY and emulator must be resized together on `tea.WindowSizeMsg`
