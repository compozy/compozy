# ADR-003: Composer Simulation for Initial Prompt

## Status

Accepted

## Date

2026-03-31

## Context

Looper prepares detailed prompts for each job (task content, file references, skill instructions) and needs to deliver them to Claude Code at job start. Previously, prompts were piped via `cmd.Stdin` in headless mode. With interactive PTY mode, Claude Code opens its TUI composer and waits for user input. The prompt must be delivered through the composer interface.

Additionally, the full prompt files are already saved in `.tmp/` by the plan phase. The composer message only needs to direct the agent to read the file — not contain the entire prompt.

## Decision

Use composer simulation: after PTY start, poll the VT emulator screen every 200ms until the Claude Code composer is detected as ready (prompt indicator like `>` at the last non-empty line). Fallback to sending after 15 seconds if detection fails. Then send a short one-liner via PTY keystroke injection: `"Read and execute the task described in /path/to/.tmp/batch-001.md"`.

Mode-specific instructions, skill references, and behavioral constraints are delivered via `--system-prompt` flag (see ADR-004), not through the composer message.

## Alternatives Considered

### Alternative 1: --prompt / -p flag

- **Description**: Pass the prompt via Claude Code's `-p` flag as a CLI argument
- **Pros**: Simple, no timing issues, no readiness detection needed
- **Cons**: The `-p` flag implies `--print` mode (headless) — Claude Code processes the prompt and exits, destroying the interactive TUI experience
- **Why rejected**: Fundamentally incompatible with interactive terminal embedding

### Alternative 2: Stdin pipe into PTY

- **Description**: Write prompt bytes directly to PTY stdin immediately after start
- **Pros**: No readiness detection needed
- **Cons**: Claude Code may not have initialized its composer yet; bytes arrive before the TUI is ready, causing corruption or partial reads; conflicts with PTY terminal setup sequences
- **Why rejected**: Race condition between PTY initialization and data delivery is unreliable

### Alternative 3: Fixed sleep (as in agh POC)

- **Description**: Wait 10 seconds after PTY start, then send via keystroke simulation
- **Pros**: Simple, proven in POC
- **Cons**: Always waits 10 seconds even if Claude Code is ready in 3 seconds; breaks if Claude Code takes longer than 10 seconds to start (slow network, heavy model)
- **Why rejected**: VT screen polling achieves the same reliability with faster startup and no arbitrary timing

## Consequences

### Positive

- Deterministic: readiness detected from actual screen state, not arbitrary timing
- Fast: sends prompt as soon as composer is ready (often 2-5 seconds)
- Robust: 15-second timeout fallback ensures forward progress even if detection fails
- Short composer message: full prompt lives in file, composer just points to it
- Clean separation: behavioral instructions in --system-prompt, task content in file

### Negative

- Readiness detection heuristic could false-positive on loading screens
- Composer simulation is more complex than simple stdin pipe
- Keystroke injection depends on Claude Code's composer accepting literal text + Ctrl+J for newlines

### Risks

- False positive readiness: mitigated by checking last non-empty line only + minimum elapsed time (2s)
- Claude Code changes composer prompt indicator: mitigated by multiple detection patterns and timeout fallback
- Very long prompts slow to type: mitigated by keeping composer message to a single short line

## Implementation Notes

- Use `emu.String()` for screen content (plain text, no ANSI codes)
- Poll from bottom of screen upward (last non-empty line)
- Keystroke injection: literal text via `pty.Write([]byte(text))`, newlines via `pty.Write([]byte{0x0a})` (Ctrl+J), submit via `pty.Write([]byte{0x0d})` (Enter)
- The `.tmp/` prompt file is already generated by the plan phase — no new file creation needed
